## 一、背景与问题

系统当前对"一天"的判定口径不统一，存在三种并存的口径。同一笔价格记录，在不同功能里会被归到不同的"日"：

| 口径 | 用在哪 | 实现方式 |
|---|---|---|
| 用户本地日（时区感知） | 支出报告的商品支出逐日统计、Expense（交通/水电气） | 前端传 `timezone_offset`（秒），后端 `local_date_range_to_utc_range` 转换 |
| 服务器本地日 | 菜谱成本趋势的起止日与逐日遍历、今日推荐的"今天" | `datetime.now().date()`，**不接收** `timezone_offset` |
| UTC 日 | 价格记录日期展示、商品/原料最新价日期判定、sparkline 迷你折线、老 price-trend | `func.date(recorded_at)` / `recorded_at.date()`（`recorded_at` 存 UTC naive） |

根本原因：`ProductRecord.recorded_at` 存 UTC naive datetime。部分代码直接取 `.date()` 得到 UTC 日；部分用 `datetime.now().date()` 得到服务器本地日；只有支出报告显式走 `timezone_offset`。三者对同一时刻的"日归属"判定不一致。

对用户的可见影响：价格/成本趋势因采用 `as_of` 前向填充语义（取截至当日最新价），日边界差几小时通常无感；但支出报告、今日推荐、跨日时段（如东八区凌晨 0–8 点）的记录归日存在偏差，且服务器部署在不同时区时偏差更明显。

---

## 二、目标

所有"把记录归到某一天"的判定，统一以**用户本地日**为准。趋势/成本的 `as_of` 前向填充聚合语义不变，仅"日边界"改用用户本地时区来切。

### 排除（不动）

- `Expense.date` 字段——本就是本地日期，正确
- 导出文件名时间戳、`exported_at`——纯时刻，不涉及"日归属"
- API datetime 序列化——已带 `+00:00`，正确
- `date_range_utils` 工具本身——是改造的基础（仅改其参数口径）
- 数据库存储——仍 UTC naive，不改存储、不迁数据

**一句话边界**：只动"把记录判归到某一天"的逻辑，不动"存时刻/展示时刻"的逻辑。

## 四、关键决策

| 决策点 | 选择 | 理由 |
|---|---|---|
| 聚合策略 | **丙·按场景混合** | 逐日查询走 `local_date_range_to_utc_range` 切 SQL 区间；多记录归日走 Python 层 `utc_datetime_to_local_date`；`as_of` 前向填充不变。三库一致、性能不退化、复用现成工具 |
| 时区表达 | **IANA 时区名**（`zoneinfo`） | 比秒偏移更准（正确处理夏令时切换日）、更标准。SQL 层不碰时区函数，绕开三库差异雷区 |
| 传递载体 | **`X-Timezone` 请求头** | 时区是客户端环境元数据，走 header 合惯例；所有请求天然统一带；URL 干净 |
| 检查点 | **`Depends(get_timezone)`** 挂涉及时区的端点 | 精准，健康检查/登录/写操作不受影响 |
| 默认时区 | **UTC**（兜底） | 中立基准。前端强制注入，正常流程几乎不触发 |
| 向后兼容 | **不兼容**旧 `timezone_offset` | 单一契约。前端同步切，项目无外部消费者 |
| 老 price-trend | **删除** | 前端未调用，YAGNI |

## 五、整体架构

```
浏览器
  │  Intl.DateTimeFormat().resolvedOptions().timeZone  →  "Asia/Shanghai"
  ▼
前端 axios 请求拦截器（api/client.ts，统一注入）
  │  所有请求带  X-Timezone: Asia/Shanghai
  ▼
后端端点
  │  tz: str = Depends(get_timezone)   →  校验非空 + 合法 IANA，否则 400
  ▼
service 层（透传 tz 字符串）
  │  ZoneInfo(tz) 解析，复用改造后的 date_range_utils
  ├─ 逐日场景 → local_date_range_to_utc_range(date, date, tz) → UTC 区间，SQL 只做 >= / <=
  └─ 归属日分组 → utc_datetime_to_local_date(recorded_at, tz) → 本地 date，Python 层分组
  ▼
数据库（不改，仍 UTC naive 存储）
```

**核心不变量**：SQL 层永远只比较 UTC 时刻（区间由 Python 层算好），不碰任何 SQL 时区函数（`func.date`、`CONVERT_TZ`、`AT TIME ZONE` 等）→ 三库安全。归属日判定全在 Python 层用 `zoneinfo` → 三库一致、夏令时正确。

## 六、`date_range_utils` 与检查点改造

### 6.1 `backend/app/utils/date_range_utils.py`

参数 `user_timezone_offset: Optional[int]` → `tz: str`（IANA 名），默认 `"UTC"`；内部 `timezone(timedelta(seconds=...))` → `ZoneInfo(tz)`。`ZoneInfo` 自带缓存，重复解析无开销。

```python
from zoneinfo import ZoneInfo
from datetime import datetime, timezone, date

def local_date_to_utc_range(local_date: date, tz: str = "UTC"):
    zone = ZoneInfo(tz)
    local_start = datetime.combine(local_date, datetime.min.time(), tzinfo=zone)
    local_end   = datetime.combine(local_date, datetime.max.time(), tzinfo=zone)
    # 数据库存 naive UTC，故剥掉 tzinfo
    return (local_start.astimezone(timezone.utc).replace(tzinfo=None),
            local_end.astimezone(timezone.utc).replace(tzinfo=None))

def local_date_range_to_utc_range(start_date: date, end_date: date, tz: str = "UTC"):
    s, _ = local_date_to_utc_range(start_date, tz)
    _, e = local_date_to_utc_range(end_date, tz)
    return s, e

def utc_datetime_to_local_date(utc_dt: datetime, tz: str = "UTC") -> date:
    if utc_dt.tzinfo is None:
        utc_dt = utc_dt.replace(tzinfo=timezone.utc)   # 数据库读出的 naive 视作 UTC
    return utc_dt.astimezone(ZoneInfo(tz)).date()
```

### 6.2 `backend/app/api/deps.py`（新建）

```python
from fastapi import Header, HTTPException
from zoneinfo import ZoneInfo, ZoneInfoNotFoundError

def get_timezone(x_timezone: str = Header(..., alias="X-Timezone")) -> str:
    """校验请求头 X-Timezone 为合法 IANA 时区名，返回时区字符串。"""
    try:
        ZoneInfo(x_timezone)
    except (ZoneInfoNotFoundError, ValueError):
        raise HTTPException(status_code=400, detail=f"无效时区: {x_timezone}")
    return x_timezone
```

涉及时区的端点声明 `tz: str = Depends(get_timezone)`，往 service 透传。

### 三条通用改法（贯穿所有场景）

1. **函数签名统一加 `tz: str`**，端点从 `Depends(get_timezone)` 拿，往 service 透传。
2. **`datetime.now()` 全换** `datetime.now(timezone.utc)` 再 `utc_datetime_to_local_date(..., tz)`——彻底消除对服务器本地时区的依赖（现 `datetime.now().date()` 暗扣服务器时区，是 bug 根源之一）。
3. **删旧链路**：前端 `utils/timezone.ts` 的 `getTimezoneOffsetSeconds`/`getTimezoneOffsetHours`、`api/timezone.ts` 的 `addTimezoneOffset`/`addTimezoneToConfig`、后端 `date_range_utils` 的 offset 形参、`report_service` 的 `user_timezone_offset` 参数——全删，不兼容。

2. **删** `utils/timezone.ts` 的 `getTimezoneOffsetSeconds` / `getTimezoneOffsetHours`。

3. **删** `api/timezone.ts` 整个文件（`addTimezoneOffset`/`addTimezoneToConfig` 全是自引用 dead code，前端无实际调用）。
