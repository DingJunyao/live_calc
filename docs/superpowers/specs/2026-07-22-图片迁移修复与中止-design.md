# 图片迁移修复与中止功能设计

日期：2026-07-22
范围：后台「图片存储配置」向导的「迁移现有图片」步骤（S3↔本地双向异步迁移）

## 背景

「切换存储后端」向导第 4 步「迁移现有图片」在 S3→本地方向存在三个问题：

1. **进度计数不实时 + 缺中止**：已上传 / 跳过 / 失败的数量要等整个迁移结束才出现；无法中止正在运行的迁移。
2. **路径未限定**：from_s3 方向把整个 bucket 的对象都列出来迁移，而非限定到应用命名空间（base_path）。
3. **迁移全部失败**：to_local 方向每个文件都失败，日志形如 `to_local 失败 livecalc/recipes/麻婆豆腐_0.jpeg: livecalc/recipes/麻婆豆腐_0.jpeg`（错误信息就是 key 本身）。

## 根因

### 问题 3（全部失败）：base_path 被加了两次

`_migrate_from_s3_core`（`backend/app/services/storage/migrate.py:40-72`）链路：

1. 用**裸 client** `s3_backend.client.list_objects_v2()` 列 bucket，返回的 `obj["Key"]` 是 bucket 里的**真实完整 key**（含 base_path 前缀），如 `livecalc/recipes/麻婆豆腐_0.jpeg`
2. 把真实 key 传给 `s3_backend.get(key)`
3. `S3Backend.get`（`backend/app/services/storage/s3.py:104`）内部 `_full_key(key)` = `f"{base_path}/{key}"` → `livecalc/livecalc/recipes/麻婆豆腐_0.jpeg`（**前缀翻倍**）
4. get_object 请求不存在的翻倍路径 → 404 → 转 `FileNotFoundError(key)`（`s3.py:107`）
5. `FileNotFoundError(key)` 的 `str()` 就是 key 本身
6. `logger.warning("to_local 失败 %s: %s", key, e)`（`migrate.py:69`），`str(e)` = key → 日志前后两个值相同，形成「失败 key: key」的迷惑形态

to_s3 方向无误（本地 walk 出逻辑 key，put 加一次 base_path 正确），仅 to_local 坏。

### 问题 2（路径未限定）

`list_objects_v2` 不带 `Prefix`（`migrate.py:47`），列出整个 bucket 的所有对象。

### 问题 1（计数不实时）

前端 `migrationStats` 读 `task.stats`（`frontend/src/views/admin/StorageConfigView.vue:668`），而 `task.stats` 仅在任务结束时写入（`backend/app/services/importer/api_service.py:107`）。运行中 `progress` 只含 `{stage, current, total, message}`（`api_service.py:55`），无 uploaded/skipped/failed 分类计数。

### 问题 1（无中止）

后台线程（`api_service.py:32-37`）无取消标志，循环无检查点。现有 `POST /import/task/{id}/cancel`（`backend/app/api/import_api.py:185`）仅置 status，后台不查（其注释自承「后台线程会继续运行，只标记状态」，对 migrate 是「假取消」）。

## 设计

核心思路：顺着项目已有的 `status="cancelled"` + 后台循环检查范式（USDA 翻译任务 `import_api.py:434` 已采用），最小波及、不加表字段。

### 改动 1：from_s3 路径限定 + base_path 剥前缀（修问题 2、3）

`_migrate_from_s3_core` 改造：

- list 带 `Prefix = base_path + "/"`（base_path 非空时），只列应用命名空间；base_path 为空列整个 bucket（专用 bucket 场景）
- list 出的真实 key 剥掉 base_path 前缀还原逻辑 key（`livecalc/recipes/a.jpg` → `recipes/a.jpg`）；剥前缀用安全判断（确认以 `base_path/` 开头才剥，否则原样）
- 用逻辑 key 调 `s3_backend.get(逻辑key)`（内部 `_full_key` 加一次 base_path → 正确的 `livecalc/recipes/a.jpg`）
- 落地 `images_dir / 逻辑key`，与 to_s3 方向（本地 `recipes/a.jpg` → put 成 `livecalc/recipes/a.jpg`）完全对称、幂等闭环
- 过滤目录占位符（key 以 `/` 结尾的跳过）

base_path 从 `s3_backend.base_path` 取（构造时已 strip `/`）。

### 改动 2：进度实时分类计数（修问题 1-A）

- `make_progress_callback`（`api_service.py:42-67`）的 callback 加可选 `counts: dict | None = None` 参数；写 progress 时若 counts 非 None 则合并进 progress dict
- `_migrate_to_s3_core` / `_migrate_from_s3_core` 循环里调 `progress_callback(stage, i+1, total, key, counts={"uploaded":..., "skipped":..., "failed":...})`
- 前端 `migrationStats` 改成优先读 `progress.uploaded/skipped/failed`（实时），回落 `stats`（结束才有）
- importer 的 `import_all` 不传 counts（默认 None，不写），向后兼容，零波及

### 改动 3：中止功能（修问题 1-B）

- 复用现有 `POST /import/task/{task_id}/cancel`（不改端点；它已能把 status 置 `cancelled`）
- `make_progress_callback` 写 progress 时顺便读 `task.status`，callback **返回** `(t.status == "cancelled")` 作为取消信号
- 两个 core 循环检查该返回值，True 则 `break`（中止延迟 ≤ 一个文件的处理时间）
- `_run_import_task`（`api_service.py:97-109`）末尾：重查 task，若 status 已是 `cancelled` 则**保留 cancelled**（不被 success/failed 覆盖），progress 记「已取消」，stats 记部分统计
- 前端进度区加「中止迁移」按钮（调 cancel 端点）；轮询到 status=`cancelled` 停表、显示部分统计

### 不做的（YAGNI）

- 不加 `cancelling` 中间态（现有 `cancelled` + 循环检查足够）
- 不加表字段（用 status，无需 alembic / 三引擎 SQL）
- 不做「暂停 / 继续」（迁移幂等，中止后再点「开始迁移」自动跳过已迁移项，等效续传）
- 不做手动子路径过滤（自动限定 base_path 已满足需求）
- 不改 to_s3 的 base_path 逻辑（其本身正确，仅需加 counts + cancel）

## 涉及文件

后端：
- `backend/app/services/storage/migrate.py` — from_s3 路径限定 / 剥前缀；两个 core 加 counts + cancel 检查
- `backend/app/services/importer/api_service.py` — `make_progress_callback` 加 counts + 返回 cancel；`_run_import_task` 保留 cancelled

前端：
- `frontend/src/views/admin/StorageConfigView.vue` — `migrationStats` 读 progress 实时计数；加「中止」按钮；轮询处理 `cancelled`

cancel 端点 `backend/app/api/import_api.py:185` 复用不改。

## 边界与一致性

- base_path 为空：list 不带 Prefix，key 不剥前缀（逻辑 key = 真实 key），get 不加前缀——正确
- base_path 非空：list 带 Prefix 限定，剥前缀，get 加一次前缀——正确
- 取消时序：cancel 端点立刻置 cancelled，后台循环在下一个文件的 progress_callback 检查到才 break，期间前端可能短暂看到 cancelled 但 progress 仍在动（与 USDA 翻译任务行为一致，可接受）
- 取消后 `_run_import_task` 末尾保留 cancelled、不覆盖；部分统计写入 stats，前端可见「取消时已迁移多少」
