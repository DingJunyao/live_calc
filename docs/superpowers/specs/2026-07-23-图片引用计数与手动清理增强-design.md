# 图片引用计数追踪与手动清理增强

## 概述

统一图片清理逻辑为手动模式，去掉编辑入口的自动删除（auto-delete），引入 `image_tracking` 表实现引用计数 + 最后使用时间追踪，并增强「未使用图片清理」管理页面的分组显示与空间统计。

## 动机

S3 和本地存储对图片的清理逻辑不统一——auto-delete 只覆盖了部分编辑路径（菜谱 PUT），且不符合系统整体的软删除/留痕风格。改为全手动清理模式，所有图片删除集中在管理页完成。

## 表结构

### image_tracking

| 列 | 类型 | 说明 |
|----|------|------|
| `key` | String(512), PK | storage key |
| `ref_count` | Integer, default=0 | 当前被引用次数 |
| `file_size` | Integer, default=0 | 文件大小（bytes） |
| `content_hash` | String(64), nullable | SHA256，用于查重 |
| `last_used_at` | DateTime, nullable | ref_count 从 1→0 的时刻 |
| `created_at` | DateTime, default=now | |
| `updated_at` | DateTime, default=now, onupdate=now | |

### 语义

- `ref_count`：该 key 被多少个业务实体引用（菜谱 images、User.avatar、ProductEntity.image_url）
- `last_used_at`：仅在 ref_count 从 1 降为 0 时写入 now()；从 0 升为正数时清 NULL
- 多个菜谱引用同一张图 → ref_count > 1

### 每次系统启动时自动扫描重建

系统启动（lifespan）时自动执行一遍扫描重建 `image_tracking` 表，
确保引用计数始终准确，不依赖事件驱动路径不出漏洞。

扫描失败不阻断启动（try/except + log warning），保证服务可用性不受影响。

资源开销（按当前项目规模估算）：
- 遍历 storage 文件：Local < 100ms / S3 约 1s（5000 张图以内）
- 引用源查询：3 张表全表扫描，< 50ms
- 总计 1~3 秒，对启动时间影响可忽略

## 引用追踪 helper

统一函数 `update_image_refs(db, old_keys, new_keys, file_sizes=None)`：

- `old_keys` 与 `new_keys` 的差集计算增减
- 增加引用：rows 不存在则创建，ref_count +1
- 减少引用：ref_count -1；减后为 0 → 记 `last_used_at = now()`
- file_sizes 可选，新建行时填充

## 引用变更入口（+1 / -1 的路径）

| 端点 | old 来源 | new 来源 | 改动说明 |
|------|---------|---------|---------|
| `PUT /recipes/{id}` | 更新前 recipe.images | 更新后的 images | 删 auto-delete，调 update_image_refs |
| `DELETE /recipes/{id}` | recipe.images | 空集 | 删图不删文件，调 update_image_refs |
| `PUT /auth/me/avatar` | 旧 avatar | 新 avatar | 新增 update_image_refs 调用 |
| `PUT /products/{id}`（含 image_url）| 旧 image_url | 新 image_url | 新增 update_image_refs 调用 |
| `POST /recipes/{id}/images` | 上传前已有 images | 上传后 images | 新增 update_image_refs 调用 |
| 数据导入批量写入 | 空集 | 写入的引用 key | 批处理调 update_image_refs |
| `POST /admin/images/scan` | — | — | 全量重建（兜底） |

## 去掉 auto-delete

删除 `backend/app/api/recipes.py` 第 547-563 行的 `# 处理 images 变更：删除已移除的图片文件（仅直写分支）` 代码块。文件清理只走「未使用图片清理」管理页。

## 图片查重（内容去重）

上传时算 SHA256 记录 `content_hash`：

- 仅记录不压缩物理存储（兼容老数据 + S3/Local 差异）
- 管理页展示形如「有 N 张图片内容相同」
- 未来物理去重可基于此字段迁移

## 扫描端点 `POST /admin/images/scan`

功能：
1. 遍历 storage 所有图片文件（`recipes/`, `avatars/` 子目录），记录 `file_size`
2. 遍历所有引用源（Recipe.images、User.avatar、ProductEntity.image_url），构建 `{key → count}` 映射
3. 合并写入 `image_tracking`：全量覆盖 `ref_count`，增量补行
4. 对于 ref_count 变为 0 的行，若 `last_used_at` 为 NULL 则记 `now()`
5. 对于 ref_count 从 0 变为正数的行，清 NULL `last_used_at`
6. 返回统计：总数、已用数、未用数、已用空间、未用空间

## 未使用图片查询 `GET /admin/images/unused`（增强）

改造为读 `image_tracking` 表 + 排除当前引用。返回分组数据：

| 分组 | 条件 | 排序 |
|------|------|------|
| 从未引用 | `ref_count = 0 AND last_used_at IS NULL` | 按 created_at 升序 |
| 180 天以上 | `last_used_at < now - 180d` | 按 last_used_at 升序 |
| 90~180 天 | `now - 180d ≤ last_used_at < now - 90d` | 按 last_used_at 升序 |
| 60~90 天 | `now - 90d ≤ last_used_at < now - 60d` | 按 last_used_at 升序 |
| 30~60 天 | `now - 60d ≤ last_used_at < now - 30d` | 按 last_used_at 升序 |
| 30 天内 | `last_used_at ≥ now - 30d` | 按 last_used_at 升序 |

每组内按 age 从老到新排列。每组统计图片张数和总空间占用。
"从未引用"始终排在最前。

## 前端增强

### UnusedImagesView.vue 改造

打开页面时自动：
1. 调 `POST /admin/images/scan`（显示 scanning loading）
2. 调 `GET /admin/images/unused` 加载分组数据
3. 按分组展示

布局：

```
┌─── 已用: 142 张 · 45.3 MB  |  未用: 38 张 · 12.7 MB ───┐
├── [未引用] 从未引用 (8 张 · 3.5 MB) ────────── [选择本组] ┤
│  img_orphan.jpg  img_test.jpg  ...                       │
├── 180 天以上不再使用 (12 张 · 4.1 MB) ───────── [选择本组] ┤
├── 90~180 天 (10 张 · 3.2 MB) ───────────────── [选择本组] ┤
├── 60~90 天 (5 张 · 1.8 MB) ────────────────── [选择本组] ┤
├── 30~60 天 (3 张 · 0.9 MB) ────────────────── [选择本组] ┤
├── 30 天内 (0 张 · 0 B) ────────────────────── [选择本组] ┤
├──────────────────────────────────────────────────────────┤
│  已选 15 张 · 4.2 MB                    [删除所选]       │
└──────────────────────────────────────────────────────────┘
```

- 分组可折叠
- 「选择本组」快速全选/取消
- 每组内图片按 key 排序
- 空间单位自动换算 B/KB/MB/GB
- 删除流程保留现有确认对话框 + 进度反馈 + 删除后前端更新
