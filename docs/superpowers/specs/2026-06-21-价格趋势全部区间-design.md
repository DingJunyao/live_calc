# 价格趋势「全部」区间标签 — 设计文档

> 日期：2026-06-21
> 范围：菜谱、商品、原料三个详情页的价格趋势图，在「周 / 月 / 季 / 年」之后追加「全部」标签，可查看完整历史区间的价格趋势。

## 1. 背景与目标

三个详情页（菜谱、商品、原料）共用公共组件 `PriceTrendChart.vue`，现有时间区间标签为「周 / 月 / 季 / 年」，组件内部按时间窗口对传入数据切片显示。

本次需求：在标签栏末尾追加「全部」，点击后显示该实体从最早记录到现在的完整价格/成本趋势。

附带修正一处现状问题：商品与原料的图表数据当前基于**分页的** `priceRecords`（每页 10 条）聚合，翻页时图表会跟着变，且无法反映完整历史。本次一并改为按**时间区间**取数。

## 2. 现状分析

### 2.1 公共组件 `frontend/src/components/charts/PriceTrendChart.vue`

- `filters`（约 L141-L146）定义四个标签：`周/月/季/年`，`value` 为 `'week'/'month'/'quarter'/'year'`，默认选中 `'month'`。
- `chartData` computed（约 L152-L176）：根据 `selectedFilter` 计算 `startDate`（周=7天、月=1月、季=3月、年=1年），对 `props.data` 做 `.filter(d => new Date(d.date) >= startDate)` 切片后排序。
- `emit('filter-change', filter)`（约 L129-L131、L193-L202、L402-L411）：切换标签时通知父组件。
- **冗余问题**：`watch(selectedFilter)` 出现了两次（L193-L202 与 L402-L411），逻辑完全重复。
- 图表用 ECharts 渲染，x 轴为日期、y 轴为价格/成本，含平均值线与 min-max 置信区间带。

### 2.2 菜谱 `frontend/src/views/recipes/RecipeDetail.vue` + 后端

- 前端维护 `costHistoryRecords`（累积数据池）与 `maxDaysLoaded`（已加载到的天数）。
- `loadCostHistory(days, offsetDays)`（约 L784-L804）调 `GET /recipes/{id}/cost-history-range`，传 `days` 与 `offset_days`，合并进池并更新 `maxDaysLoaded`。
- `enqueueCostHistory(targetDays, showLoading)`（约 L807-L832）：串行队列，按 `targetDays` 计算剩余天数一次性请求。
- `onCostTrendFilterChange(filter)`（约 L841-L855）：将 `week/month/quarter/year` 映射为 7/30/90/365 天，若超出 `maxDaysLoaded` 则入队加载。
- `chartData` computed（约 L858-L866）：基于 `costHistoryRecords` 按当前份数折算。
- 后端 `GET /recipes/{id}/cost-history-range`（`backend/app/api/recipes.py` 约 L791-L842）：`days: Query(90, ge=7, le=365)`、`offset_days: Query(0, ge=0)`。
- 后端 `calculate_recipe_cost_range_trend`（`backend/app/services/recipe_service.py` 约 L1181+）：批量预加载商品/价格记录/回退/包含关系后**逐天遍历**计算成本。`start_date = max(earliest_record, end_date - days)`，内部已用最早价格记录日期兜底钳制。成本趋势按天前向填充，**每天都有值，无日期空档**。

### 2.3 商品 `frontend/src/views/products/ProductDetail.vue` / 原料 `frontend/src/views/ingredients/IngredientDetail.vue`

- `priceRecords`（商品 L1179、原料 L1772）为**分页数据**：`pricePageSize = 10`，`loadPriceRecords` 带 `skip/limit`，`watch(pricePage, loadPriceRecords)` 翻页。
- `chartData` computed（商品 L1844-L1885、原料 L2583+）基于 `priceRecords.value` 聚合：按日期分组，归一化单位（商品归一化到 ¥/斤；原料转换到原料默认单位），算 min/max/avg/count。
- 两个详情页的 `<PriceTrendChart>` **未绑定** `@filter-change`（组件内部切片即可）。
- 后端 `GET /products`（`backend/app/api/products.py` 约 L231-L248）：已支持 `start_date` / `end_date` 可选参数（`Optional[datetime]`，按 `recorded_at` 过滤，三个查询分支均已实现），**前端可直接复用，无需后端改动**。

### 2.4 10s 超时约束

- `frontend/src/api/client.ts` L6：axios 实例 `timeout: 10000`。
- axios 支持**单请求覆盖**：`api.get(url, { timeout: 30000 })` 可单独放宽某次请求，无需改全局。
- 风险评估：
  - **菜谱**：重计算（批量预加载 + 逐天遍历），「全部」跨度大时单请求易超 10s。但成本趋势每日有值，分批终止可靠。
  - **商品/原料**：轻量 SQL 查询，典型 <1s。但原始价格记录**有日期空档**（某年有、某年空、再前又有），分批「空批终止」会误判漏数据。

## 3. 设计方案

### 3.1 公共组件 `PriceTrendChart.vue`（改这一份，三处共用）

1. `filters` 末尾追加 `{ label: '全部', value: 'all' as const }`。
2. `selectedFilter` 的类型联合改为 `'week' | 'month' | 'quarter' | 'year' | 'all'`；`emit('filter-change')` 的类型同步加上 `'all'`。
3. `chartData` computed 增加 `'all'` 分支：**不计算 `startDate`、不做时间过滤**，直接返回 `props.data` 全部（仅保留排序）。
4. 清理重复的第二个 `watch(selectedFilter)`（L402-L411，与 L193-L202 重复）。

### 3.2 菜谱：分批拉取（后端上限不改）

后端 `days` 上限**保持 `le=365` 不变**（不放宽），改用前端分批规避超时。

前端 `RecipeDetail.vue`：

- `onCostTrendFilterChange` 增加 `'all'` 分支，调用一个新的 `loadAllCostHistory()`：
  - 循环：每批 `loadCostHistory(365, maxDaysLoaded.value)`，`offset_days` 随 `maxDaysLoaded` 自然递增。
  - 合并进 `costHistoryRecords`（现有逻辑已处理合并与 `maxDaysLoaded` 更新）。
  - **终止条件**：某批返回空记录（`costHistoryRecords.length` 不再增长）即停——菜谱每日有值，空批表示已早于最早记录，终止可靠。
  - 全程 `loadingCostHistory = true`，结束后置 false。
- 组件层 `selectedFilter === 'all'` 时不切片，显示累积池全部。

> 每批 `days=365` 在后端为预加载固定开销 + 365 天遍历，单批远低于 10s。数据跨度越长批数越多，但趋势图随每批合并渐进刷新。

### 3.3 商品 / 原料：按时间区间取数 + 累积池 + 「全部」放宽超时

#### 后端

`GET /products` 已支持 `start_date` / `end_date`（见 2.3），**后端零改动**。前端「全部」不传 `start_date` 即取全量；常规区间传 `start_date=YYYY-MM-DD`（FastAPI 自动解析为当天 00:00 的 datetime）。

#### 前端（`ProductDetail.vue` 与 `IngredientDetail.vue` 各加一份等价逻辑）

- 新增图表专用累积池 `chartPriceRecords`，与「已请求覆盖到的最早日期」标记 `chartEarliestDate`（`undefined`=未请求、`Date`=已覆盖到该日、`null`=已全量）。
- 新增 `loadChartPriceRecords(startDate?)`：
  - 计算是否需要请求：若 `startDate` 非空且 `chartEarliestDate` 已是更早的日期或 `null`，则直接返回（已覆盖，不发请求）。
  - 请求 `GET /products`，带 `product_id`（商品）或 `ingredient_id`（原料），带 `start_date`（「全部」时不传），`limit` 传保护值。
  - **「全部」请求（`startDate` 为空）单独把 `timeout` 放宽到 30000ms**；常规区间维持默认 10s。
  - 按 id 去重合并结果到 `chartPriceRecords`；更新 `chartEarliestDate`（「全部」置 `null`，否则取 `min(当前, startDate)`）。
- 给 `<PriceTrendChart>` 绑定 `@filter-change="onPriceTrendFilterChange"`：
  - `week/month/quarter/year` → 计算 `startDate`（今天往前 7/30/90/365 天），调 `loadChartPriceRecords(startDate)`。
  - `all` → 调 `loadChartPriceRecords()`（不传 `startDate`，触发全量 + 放宽超时）。
- 首次加载详情时默认拉「月」：`loadChartPriceRecords(today - 30d)`，与组件默认选中 `'month'` 对齐。
- `chartData` computed 的聚合逻辑（按日分组、单位归一化、min/max/avg/count）**原样保留**，仅将数据源由 `priceRecords.value` 改为 `chartPriceRecords.value`。
- 下方分页价格记录列表逻辑（`priceRecords` / `pricePage` / `loadPriceRecords`）**完全不动**。
- 新增独立的图表加载态标记（如 `loadingChartPrices`），传给组件 `:loading`，避免与列表的 `loadingPrices` 互相干扰。

> 累积池保证：切过「全部」之后再切回年/季/月/周，`chartEarliestDate === null`，不发请求，组件直接切片；与菜谱体验一致。常规区间单次请求快（区间内记录少），不放宽超时；仅「全部」放宽兜底，且因原始记录有日期空档**不分批**。

### 3.4 三实体策略对照

| 实体 | 「全部」取数 | 超时规避 | 后端改动 |
|------|------------|---------|---------|
| 菜谱 | 分批循环，每批 `days=365`，空批止 | 分批，每批 <10s | 无（`le=365` 保持） |
| 商品 | 单请求全量（不传 `start_date`） | 该请求 `timeout=30s` | 无（端点已支持） |
| 原料 | 单请求全量（不传 `start_date`） | 该请求 `timeout=30s` | 无（同上） |

## 4. 数据流

- **菜谱**：`onCostTrendFilterChange('all')` → `loadAllCostHistory()` 循环分批 → 合并 `costHistoryRecords` → `chartData` computed 折算份数 → 组件 `selectedFilter='all'` 不切片 → ECharts 渲染。
- **商品/原料**：`onPriceTrendFilterChange(filter)` → `loadChartPriceRecords(startDate)` → 合并 `chartPriceRecords` → `chartData` computed 聚合 → 组件按 `selectedFilter` 切片（`all` 不切） → ECharts 渲染。

## 5. 边界与取舍

- **商品/原料「全部」的极端数据量**：`limit` 作为上限保护（默认上限 1000）。若某实体记录超过该上限，会被截断；属极端情况，可接受。
- **菜谱分批粒度**：每批 365 天。若用户数据跨度极大（如十年），批数较多但每批仍快、且渐进刷新，体验可接受；不引入更复杂的动态粒度（YAGNI）。
- **「全部」是否设为默认**：不设。默认仍为「月」，与现状一致。
- **商品/原料不改分页列表行为**：下方的价格记录分页列表保持现状，仅图表数据源独立化。

## 6. 验证

- 前端：`npm run build` 通过（项目硬要求）。
- 后端：零改动，无需验证（`/products` 已支持 `start_date/end_date`）。
- 手动验证三个详情页：
  - 菜谱：点「全部」分批加载完整成本历史，趋势图渐进刷新；切回年/季/月/周正常。
  - 商品：点「全部」显示完整价格历史（不受 10 条分页限制）；切回各区间正常；下方价格列表翻页不影响图表。
  - 原料：同商品。
  - 三处「全部」均不触发 10s 超时（菜谱分批、商品/原料放宽超时）。

## 7. 不在本次范围

- 不修改现有「周/月/季/年」的语义或默认值。
- 不改动商品/原料下方的分页价格记录列表。
- 不引入后端异步任务/轮询机制（当前分批 + 放宽超时已足够）。
