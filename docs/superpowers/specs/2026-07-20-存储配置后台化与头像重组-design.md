## 一、背景与目标

### 背景

当日三阶段上线后，启动验证发现多处体验缺口：

1. **个人信息编辑入口分散**：头像上传是 [ProfileView.vue:53-88](../../../frontend/src/views/profile/ProfileView.vue#L53-L88) 顶部独立卡片，地区选择是独立对话框 `regionDialog`（[第 509 行](../../../frontend/src/views/profile/ProfileView.vue#L509)），都与设置列表里的「用户信息编辑」对话框（[第 93 行](../../../frontend/src/views/profile/ProfileView.vue#L93) `openAccountDialog`）分离；且昵称输入框（[第 262 行](../../../frontend/src/views/profile/ProfileView.vue#L262)）带多余 `prepend-icon`、级联第一级文案「国家」欠准。用户预期这些都在「用户信息编辑」一处。
2. **地区级联 No data + 无更新入口**：[`administrative_regions` 表实测为空](../../../backend/data/livecalc.db)（count=0），级联打开第一级就「No data available」。根因是 [main.py:259-263](../../../backend/app/main.py#L259-L263) 的 `ensure_administrative_regions` 被关在「菜谱首次初始化」`else` 块内，非首次启动的库（已有菜谱）不执行 seed。行政区划是纯地理基础数据，**初始化就该写入**；且数据维护中心缺一个手动更新入口（对齐过敏原「快速导入」按钮范式）。
3. **存储配置无后台入口**：存储配置全在 [config.py:60-72](../../../backend/app/config.py#L60-L72) 的 `Settings`（从 `.env` 读），后台无配置页。问题链：
   - 初始化导入菜谱必导图片 → 初始化时配置只能从 `.env` 读（DB 还没起来 / 没配置）
   - 后续要改配置（如 local 切 S3）→ 不应让后端改 `.env`（docker 部署 `.env` 只读）
   - 配置存 DB 才能在线改 → 与 `.env` 同名变量冲突，管理员分不清哪个生效

### 目标

1. **个人信息编辑重组**：头像、地区入口都收进「用户信息编辑」对话框（顶部头像卡、独立地区对话框整块移除）；昵称输入框去 `prepend-icon`；级联第一级文案「国家」→「国家/地区」。
2. **地区 seed 修复 + 更新模块**：`ensure_administrative_regions` 移出菜谱首次初始化条件块，无条件 ensure（初始化写入）；数据维护中心新增「行政区划」模块，手动触发 upsert 更新 / 补缺失（对齐过敏原 ensure + 按钮双轨范式）。
3. **存储配置后台化**：引入 `StorageConfiguration` DB 表，DB 覆盖 `.env`（`.env` 降级为 bootstrap baseline）；后台新增「图片存储」配置页，向导式切换 local↔S3，强制测试连接防配错锁死，**双向**迁移走异步任务。

### 非目标（YAGNI）

- 存储凭证加密存储不做（对齐 `map_config` 明文存 key 的惯例，DB 已是管理员才能访问的资源；展示侧脱敏即可）。
- 不改 `POST /auth/me/avatar` 端点本身（已就绪），只改前端入口位置。
- 不动当日已上线的 StorageBackend 接口 / LocalBackend / S3Backend / 三环节收口逻辑。

## 四、子任务二·存储配置后台化

### 4.1 配置三层优先级模型

「DB 覆盖 .env」的精确语义——**逐字段**判定来源：

```
读取优先级（逐字段）
  ① DB storage_configurations 表该字段非空  →  用 DB       source=db
  ② 否则 Settings（.env BOOTSTRAP_*）有值    →  用 .env     source=env
  ③ 都没有                                  →  用代码默认  source=default
```

- `.env` 降级为 **bootstrap baseline**：首次启动（菜谱导入图片、DB 未配置）时兜底。
- 后台一旦写入 DB，DB 接管该字段；docker 只读 `.env` 不再是在线改的障碍。
- GET 接口返回每字段 `source` 标注，管理员一眼看清「现在生效的是 DB 还是 .env」。

### 4.2 数据层：`StorageConfiguration` 表

对齐 [`MapConfiguration`](../../../backend/app/models/map_config.py) 范式，单行结构化表，独立列（全标量，不必塞 JSON）：

```python
class StorageConfiguration(Base):
    __tablename__ = "storage_configurations"
    id = Column(Integer, primary_key=True)
    backend = Column(String(10), nullable=False, default="local")       # local | s3
    storage_base_url = Column(String(255), nullable=True)               # local 端对外基址
    s3_endpoint = Column(String(255), nullable=True)
    s3_access_key = Column(String(255), nullable=True)
    s3_secret_key = Column(String(255), nullable=True)                  # 敏感，GET 脱敏
    s3_bucket = Column(String(255), nullable=True)
    s3_region = Column(String(64), nullable=True)
    s3_url_style = Column(String(10), nullable=False, default="path")   # path | virtual
    created_at = Column(DateTime, default=datetime.utcnow)
    updated_at = Column(DateTime, default=datetime.utcnow, onupdate=datetime.utcnow)
```

配套：
- helper `get_stored_storage_config(db)` / `update_stored_storage_config(db, data)`（对齐 [admin.py:86-112](../../../backend/app/api/admin.py#L86-L112) 地图配置 helper，不存在则建默认行）。
- `DEFAULT_STORAGE_CONFIG` 常量（全字段 None / `backend='local'` / `s3_url_style='path'`）。
- alembic 迁移 + SQLite/MySQL/PostgreSQL 三引擎 SQL + 开发库直接补（项目惯例，不走 alembic upgrade）。PostGIS 无关，不发 PostGIS 变体。

### 4.3 .env 变量改名（BOOTSTRAP_ 前缀）

8 个变量全部加 `BOOTSTRAP_` 前缀，明确「初始化引导用」，与后台 DB 配置彻底区分：

| 旧名 | 新名 |
|---|---|
| `STORAGE_BACKEND` | `BOOTSTRAP_STORAGE_BACKEND` |
| `STORAGE_BASE_URL` | `BOOTSTRAP_STORAGE_BASE_URL` |
| `S3_ENDPOINT` | `BOOTSTRAP_S3_ENDPOINT` |
| `S3_ACCESS_KEY` | `BOOTSTRAP_S3_ACCESS_KEY` |
| `S3_SECRET_KEY` | `BOOTSTRAP_S3_SECRET_KEY` |
| `S3_BUCKET` | `BOOTSTRAP_S3_BUCKET` |
| `S3_REGION` | `BOOTSTRAP_S3_REGION` |
| `S3_URL_STYLE` | `BOOTSTRAP_S3_URL_STYLE` |

- [config.py](../../../backend/app/config.py) `Settings` 字段同步改名：`storage_backend` → `bootstrap_storage_backend`，依此类推（共 8 个字段）。
- [.env.example](../../../backend/.env.example) 注释明确：**「以下变量仅在初始化 / 首次启动 / DB 未配置时生效；后台「图片存储」页配置后优先级更高」**。
- 影响面：改造后 `Settings` 的存储字段只在 `load_effective_storage_config()` 内作 fallback 被引用一次（factory 不再直读），改名波及面小；[migrate_local_to_s3.py](../../../backend/scripts/migrate_local_to_s3.py) CLI 脚本直读 `settings.bootstrap_s3_*`（脚本是无后台页面时的命令行兜底，走 bootstrap 配置合理）。
- 老的 `.env`（开发者本机）变量名失效，需手动改成 `BOOTSTRAP_*`；默认 `local` 模式不受影响（DB 空 → bootstrap 空 → 代码默认 `local`）。

### 4.4 factory 改造（精髓）

[factory.py](../../../backend/app/services/storage/factory.py) 从直读 `settings` 改为读 **effective 配置**，保留 lru_cache 单例 + boto3 懒加载：

```python
@lru_cache(maxsize=1)
def _build() -> StorageBackend:
    cfg = load_effective_storage_config()   # 新增：DB→.env→默认 三层逐字段合并
    if cfg.backend == "s3":
        from app.services.storage.s3 import S3Backend
        return S3Backend(endpoint=cfg.s3_endpoint, bucket=cfg.s3_bucket,
                         access_key=cfg.s3_access_key, secret_key=cfg.s3_secret_key,
                         region=cfg.s3_region, url_style=cfg.s3_url_style)
    return LocalBackend(root=_STATIC_ROOT,
                        base_url=cfg.storage_base_url or "/api/v1/static")
```

`load_effective_storage_config()`：
- 用独立 `SessionLocal()` 读 DB（无请求上下文也能跑，对齐 [security.py](../../../backend/app/core/security.py) `resolve_user_from_token` 范式），try/finally 归还。
- 逐字段判定：DB 非空用 DB，否则回落 `settings.bootstrap_*`，再否则代码默认。
- 返回带 `source` 标注的 dataclass（每字段独立 source），供 GET 接口透出。

**热生效**：`PUT /admin/storage-config` 写完 DB 后立即 `reset_storage()`（清 lru_cache），下一轮 `get_storage()` 自动重建——后台改完不重启即生效。

### 4.5 API 四端点（均 `get_current_admin_user`）

挂 `admin` router，路径前缀 `/admin/storage-config`：

| 端点 | 作用 | 关键点 |
|---|---|---|
| `GET /admin/storage-config` | 返当前 effective 配置 + 每字段 `source`（db/env/default） | `s3_secret_key` 脱敏返 `***` + `has_secret: bool` 标志；`s3_access_key` 同理脱敏 |
| `PUT /admin/storage-config` | 写 DB（upsert 单行），写完调 `reset_storage()` | `s3_secret_key=null` / `"***"` 表「沿用旧值」防误清空；**仅当 `backend=s3` 时**写前再跑一次探针（防绕过向导直调 API），切 `local` 不跑探针直接写 |
| `POST /admin/storage-config/test` | body 是待测配置（**不落库**），建临时 S3Backend，put+get+delete 一个探针对象 | 返 `{ok: bool, error?: str}`；向导「测试连接」调它 |
| `POST /admin/storage-config/migrate` | 触发双向异步迁移（`direction: to_s3`/`to_local`），返 `task_id` | 复用 `ImportTask`（`task_type='storage_migrate'`），见 4.6 |

**凭证脱敏规则**：
- GET：`s3_access_key` / `s3_secret_key` 不返原值，返 `***` + `has_access_key` / `has_secret` 标志。
- PUT：`s3_access_key` / `s3_secret_key` 传 `null` / `"***"` 表示「不变」；传明文表示「更新」。后端区分处理。
- 比 `map_config` 严（地图 key 必须明文返前端用），因为对象存储凭证是后端用、权限大，前端只需展示「已设置/未设置」。

### 4.6 异步迁移（复用 ImportTask）

**复用 [`ImportTask`](../../../backend/app/models/import_task.py) 表，不新建表**——迁移任务极低频（切 backend 才跑），`task_type` 字段本就开放：

- 迁移任务用 `task_type='storage_migrate'` 写进 `import_tasks`，stats 带 `direction: to_s3|to_local`。
- 仿 [`_run_import_task`](../../../backend/app/services/importer/api_service.py#L70) 写专门 `_run_migrate_task(task_id, direction, s3_config)`：跑迁移逻辑、stats 写 `{direction, uploaded, skipped, failed, total}`。**不复用 `_run_import_task`**——那套绑死 `ImportResult.stats/errors`，迁移不是 importer，硬塞别扭。
- **双向迁移，list 逻辑在迁移模块内各自实现，不扩 `StorageBackend` Protocol**：
  - `to_s3`：walk 本地 `backend/static/images/` → `S3Backend.put`（沿用 [`_migrate_all`](../../../backend/scripts/migrate_local_to_s3.py#L51)）。
  - `to_local`：`S3Backend` 的 boto3 client `list_objects_v2` 列举 bucket 全部 key → `get` → 落本地 `images/<key>`（新增反向逻辑）。
  - 不给 Protocol 加 `list_keys()`：迁移是运维操作，list 的语义两边差异大（本地 walk vs S3 API），内聚在迁移模块更干净。
- 给迁移函数加 `progress_callback` 参数，每传一张图回一次 `{stage, current, total, message}`（对齐导入进度回报粒度，复用 [`make_progress_callback`](../../../backend/app/services/importer/api_service.py#L42)）。
- 方向由 `POST /migrate` 的 `direction` 参数显式传（前端向导按「目标 backend」推断：当前 local→目标 s3 即 `to_s3`，反之 `to_local`）。
- `to_s3` 用**向导第②步填的待保存 S3 配置**建目标 S3Backend；`to_local` 用**当前生效 S3 配置**（effective）建源 S3Backend → `start_background_import` 起线程 → 返 `task_id`。
- 幂等：`to_s3` S3 已存在跳过（沿用）；`to_local` 本地已存在跳过（不覆盖，防回滚丢新图）。
- 前端两处展示：
  - 向导第④步内自轮询进度（用户在向导里等结果）。
  - 「数据维护中心」任务列表 `mergedTasks` 自动收入 `storage_migrate`（加展示分支：图标 + 名称「图片存储迁移」+ stats 映射 uploaded/skipped/failed）。
- CLI 脚本双向兜底：[migrate_local_to_s3.py](../../../backend/scripts/migrate_local_to_s3.py) 保留（`to_s3`，读 `settings.bootstrap_s3_*`）；新增反向 `migrate_s3_to_local.py`（`to_local`，读 effective S3 配置），供无后台页面时命令行运维。

### 4.7 后台页：「图片存储」（路由 `/admin/storage`）

- 后台导航放「地图配置」旁（同属基础设施配置），图标 `mdi-cloud-outline` / `mdi-storage`。
- 页面主体：当前 effective 配置展示卡（每项标 source chip：DB / .env / 默认）+ 「切换存储后端」按钮打开向导。

### 4.8 向导五步（local↔S3 切换流程）

```
① 选择目标 backend      → local / s3 单选
② 填写目标配置          → S3: endpoint/bucket/access_key/secret_key/region/url_style
                          local: storage_base_url
③ 测试连接（强制）      → POST /test 用②的配置建临时 backend，put/get/delete 探针
                          失败 → 阻断，不许进下一步（防配错锁死）
④ 迁移现有图片（可选）  → 目标 backend 与当前不同时出现，方向自动：
                          当前 local→目标 s3：检测本地有图，推到 S3（to_s3）
                          当前 s3→目标 local：检测 S3 有图，拉回本地（to_local）
                          POST /migrate（带 direction）起 task → 向导内轮询进度
                          可跳过（前端警告"切后源端独有图将无法访问"）
⑤ 确认切换              → PUT /storage-config 写 DB（含②配置）+ reset_storage()
                          完成提示，前端图从新 backend 加载
```

- 向导可取消，已起的迁移任务不取消（后台继续跑完，幂等无副作用）。

### 4.9 安全退路（防把自己锁死）

- **测试连接强制**：向导第③步是硬闸，S3 凭证错了根本进不到切换。
- **PUT 前再校验**：`PUT /storage-config` 后端对即将写入的 S3 配置再跑一次探针（防绕过向导直接调 API），失败回滚不写 DB。
- **secret 不变哨兵**：`PUT` 时 `s3_secret_key=null` 表示「沿用旧值」，前端脱敏展示 `***`，不会因保存别的字段误清空凭证。
- **重置按钮**：页面提供「重置为 bootstrap / 默认」——清空 DB 行，回退到 `.env` baseline（docker 场景的救命稻草）。

### 4.10 向后兼容

- `.env` / `Settings` 的 `BOOTSTRAP_*` 字段保留（baseline 必需）。
- DB 表首次为空：`load_effective_storage_config()` 查 DB 没行 → 全字段回落 `.env`，等于现状。
- 老部署不碰后台页，行为零变化；老 `.env` 变量需手动加 `BOOTSTRAP_` 前缀（dev 影响：默认 local 不受影响；配过 S3 的需改前缀）。
- [main.py StaticFiles 挂载保留](../../../backend/app/main.py#L550)：local 模式必需（LocalBackend 落盘 static，StaticFiles 对外暴露）；S3 模式挂载无害（可作 fallback）。

## 八、风险与待定

- **factory 读 DB 的启动期时序**：lifespan 启动导入菜谱图片时调 `get_storage()`，此时 DB 表须已建（`create_all` 已覆盖，开发库直补需先跑）。`load_effective_storage_config()` 读 DB 失败（表未建/异常）须 fallback 到 `.env` 不阻断启动。
- **凭证脱敏与 PUT 幂等**：前端保存时若不改凭证必须传 `null`，UI 须明确引导（占位符「未修改」），否则易误传空串清空。
- **lru_cache + DB 单例的测试隔离**：单测间须 `reset_storage()`，且 `load_effective_storage_config` 读 DB 的 mock 要到位（避免单测污染真实 DB）。
- **`BOOTSTRAP_` 改名的存量影响**：已配 S3 的老 `.env` 需手动改前缀，发布需在 changelog 显著说明。
- **PUT 前探针的网络代价**：每次保存 S3 配置都跑一次 put/get/delete，慢网络下有延迟；可接受（保存是低频操作）。
- **`to_local` 反向迁移的两个坑**：① S3 `list_objects_v2` 单页最多 1000 keys，图多须分页累积；② 拉取前须确认本地磁盘空间充足（全量图可能数百 MB～数 GB），可考虑迁移前预估总量提示。
- **地区 seed 执行时长（lifespan + 更新按钮）**：`ensure_administrative_regions` 无条件化后，首次启动空表要处理 ~3700 条；「更新」按钮走 `upsert`（按 code 逐条查 + 软删复活）比纯 insert 慢。两路径都需批量优化避免卡顿（对齐 [allergen_seed](../../../backend/app/services/allergen_seed.py) 批量范式）；lifespan 幂等保证二次启动 count>0 跳过，按钮走 upsert 幂等可重复点。
- **地区级联搬进对话框的布局**：4 级级联 + 原有字段致 `accountDialog` 变长，桌面 520px 宽 + 纵向滚动可接受；移动端需测滚动体验。
