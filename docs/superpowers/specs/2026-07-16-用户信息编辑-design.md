# 用户信息编辑 · 设计

## 背景

当前用户无法在个人中心修改自己的用户名、邮箱、密码。`PATCH /auth/me`（`backend/app/api/auth.py` 的 `update_me`）只支持营养目标、预算与单位偏好（`UserProfileUpdate`），不含任何账号身份字段。改密码的唯一途径是管理员在「用户管理」里重置。

本功能让普通用户在个人中心自助修改用户名、邮箱、手机号、密码。

## 目标

- 个人中心新增「用户信息编辑」入口（列表第一项，位于「主题切换」之上）。
- 单对话框支持改：用户名、邮箱、手机号、密码。
- 改用户名/邮箱/手机：即时查重生效，不需要密码。
- 改密码：必须先校验当前密码；改成功后用户不掉线（无缝续登）。
- 复用现有查重、SHA256 密码、token 版本机制，不引入新表。

## 非目标（YAGNI）

- 不做邮箱验证流程（改邮箱不触发验证邮件，`email_verified` 维持原值）。
- 改用户名/邮箱/手机不要求输入密码（已登录即视为可信）。
- 不做密码强度策略（沿用注册时已有的长度约定）。
- 不支持改 `is_admin` / `is_active`（那是管理员的事）。

## 后端设计

### 新增端点

`PUT /auth/me/account`（当前用户自行更新账号信息）。字段全部可选，传了才改。

请求 schema `UserAccountUpdate`（新增于 `backend/app/schemas/auth.py`）：

```python
class UserAccountUpdate(BaseModel):
    username: Optional[str] = Field(None, min_length=3, max_length=50)
    email: Optional[EmailStr] = None
    phone: Optional[str] = Field(None, pattern=r"^1[3-9]\d{9}$")
    current_password: Optional[str] = None   # 前端 SHA256 后传
    new_password: Optional[str] = None       # 前端 SHA256 后传

    @field_validator('phone', mode='before')
    @classmethod
    def empty_string_to_none(cls, v):
        if v == '' or v is None:
            return None
        return v
```

响应：`UserResponse` +（仅当改了密码时）新的 `access_token` / `refresh_token`。用一个新的响应 schema 承载：

```python
class UserAccountResponse(BaseModel):
    user: UserResponse
    access_token: Optional[str] = None
    refresh_token: Optional[str] = None
```

### 端点逻辑（`auth.py` 新增 `update_my_account`）

1. 取 `current_user`。
2. 改用户名（`username` 非空且与当前不同）：查重 `User.username == new and User.id != current_user.id`，命中则 400「用户名已被占用」。
3. 改邮箱（`email` 非空且不同）：`EmailStr` 由 schema 校验；查重同上（`User.email`），命中 400。
4. 改手机（`phone` 非空且不同）：查重（`User.phone`，忽略 NULL），命中 400。
5. 改密码（`new_password` 非空）：
   - 要求 `current_password` 非空，否则 400「修改密码需提供当前密码」。
   - `verify_password(current_password, current_user.password_hash)` 失败 → 401「当前密码错误」。
   - `current_user.password_hash = get_password_hash(new_password)`。
   - `current_user.token_version += 1`。
   - 用新 `token_version` 签发新 access + refresh token，放进响应。
6. 提交事务。`db.commit()` 后 `db.refresh(current_user)`。
7. 返回 `UserAccountResponse`（user + 可选 token）。

不改密码时，`token_version` 不动，`access_token` / `refresh_token` 为 `None`。

### 鉴权

复用 `get_current_user` 依赖（与 `PATCH /me` 一致）。无需新权限。

### 安全要点

- 改密码强制校验当前密码，防登录态被劫持时被改密码。
- 改密码触发 `token_version += 1`：旧 access/refresh token 全部失效（含其他设备）。同时签发新 token 返回给发起方，发起方不掉线；其他设备的旧 token 失效需重新登录——这是预期行为（改密后其他设备掉线符合安全预期）。
- 用户名/邮箱/手机改动不影响 `token_version`，登录态不受影响。

## 前端设计

### ProfileView 入口

在「主题切换」上方新增一项「用户信息编辑」（`mdi-account-edit`，subtitle「用户名、邮箱、密码」），点击打开对话框。该项成为设置列表第一项。

### 对话框（内联于 ProfileView，复用现有 v-dialog 模式）

分两区：

- 基本信息：用户名、邮箱、手机号（手机可清空）。
- 修改密码：当前密码、新密码、确认新密码。三框全留空 = 不改密码。

保存前前端校验：

- 用户名 3–50 字符。
- 邮箱格式。
- 手机号格式（可空）。
- 若填了新密码：当前密码必填、新密码长度符合约定、新密码 = 确认。

保存：

- 密码字段按现有 Register/Login 的方式做 SHA256 后再提交（复用同一 hash 工具函数）。
- 调 `PUT /auth/me/account`，请求体只带「有变化」的字段；密码区全空则不带 `current_password` / `new_password`。
- 成功：
  - 若响应含新 token → `userStore` 写入新 access/refresh token（localStorage + axios 头）。
  - `userStore.fetchUser()` 刷新用户信息（用户名等会变）。
  - 全局 snackbar 提示「已更新」（改了密码则补一句「登录态已刷新」）。
- 失败：读 `error.userMessage`，显示后端具体原因（用户名占用、当前密码错误等）。

## 测试

后端（`backend/tests/`，参考现有 auth 测试写法）：

- 只改用户名：成功；用户名重复 → 400；不改密码时响应不含 token。
- 改邮箱重复 → 400；改手机重复 → 400。
- 改密码：当前密码错 → 401；当前密码对 → 成功，响应含新 token，`token_version` +1，旧 token 失效。
- 改密码未带 `current_password` → 400。
- 改密码后用新 token 可访问、用旧 token 失效。

前端：`npm run build` 通过。
