# 菜谱发布默认私有 · 设计文档

- 日期：2026-07-09
- 分支：master（主分支直接修改）
- 实现方式：subagent-driven-development

## 一、问题

需求：用户创建菜谱默认是私有的，只有点菜谱详情的「发布」按钮才发布；发布是不可逆操作；菜谱详情导航栏对未发布菜谱提供发布按钮。

现状与需求不符——用户一创建菜谱就处于「已发布/公开」状态。

## 二、根因

`source` 字段被赋予了两重语义：①菜谱来源标识（`custom`/`json_repo`/`import`/`howtocook:xxx`）；②是否公共/已发布。共享转型引入 `is_public` 本是为了分离这两个语义，但未清理干净 `source` 的「公共」语义残留。

具体地：

- [recipe.py:27](backend/app/schemas/recipe.py#L27) `RecipeCreate.source` 默认值 `"custom"`，用户自创菜谱的 `source` 是个**非空字符串**。
- 全系统把「`source` 非空」当成「公共/已发布」：
  - 列表公共分支 [recipes.py:181](backend/app/api/recipes.py#L181) `or_(Recipe.source != None, Recipe.is_public == True)`
  - 前端 [RecipeDetail.vue:559](frontend/src/views/recipes/RecipeDetail.vue#L559) `isPublished = is_public || !!source`
- 所以用户刚建好的菜谱 `source='custom'` 非空 → 立刻被判「已发布」、对所有人可见、发布按钮（`canPublish = !isPublished`）永不出现。
- 此外 [recipes.py:111-113](backend/app/api/recipes.py#L111) 管理员创建即 `is_public=True`，是叠加在上述根因之上的次要问题。

附带隐患：详情可见性 [recipes.py:378-389](backend/app/api/recipes.py#L378) 只看 `source != None`，意味着**别人自创的私有菜谱详情其实能被看到**（`source='custom'` 非空命中），是个权限漏点。

## 三、方案

**方案 A（已选定）：彻底分离语义——「公共/已发布」一律以 `is_public` 为唯一准绳，`source` 回归「来源标识」本职，不再参与任何可见性/发布判断。**

未选方案 B（把 `source` 默认值改为 `None`），因其仍把「公共」语义挂在 `source` 上，治标不治本。

### 3.1 语义定调

- 自创菜谱：创建即 `is_public=False`（私有），点发布按钮才 `is_public=True`。
- 导入菜谱（howtocook / json_repo / 导入导出）：创建即 `is_public=True`（公共知识库）。
- 「公共/已发布」判断一律 `is_public`，`source` 纯标识。
- 发布仍走 change_proposals 审核框架：普通用户 `submit` 待审、管理员 `apply_as_admin` 直写。
- **作者发布后不可撤回**：作者侧无任何「取消发布」入口（天然不可逆）；管理员审核台保留执行器 `revert` 作为救场（万一误发布）。

#### 数据迁移（必要，非可选）

判断逻辑一旦只看 `is_public`，现有导入菜谱（`source` 非空但 `is_public=False`）会瞬间变私有，菜谱列表大面积消失。必须先把它们 `is_public` 置 `true`：

```sql
UPDATE recipes SET is_public = true
WHERE source IS NOT NULL AND source <> 'custom' AND is_public = false;
```

- 存量 `source='custom'` 菜谱**不动**（符合「存量不动」决策）——方案 A 下它们靠 `is_public` 判定：普通用户创建的 `is_public=False` 自动回归私有，管理员创建的 `is_public=True` 保持公开。
- 产物：alembic 迁移 + 三引擎 SQL 脚本（SQLite / MySQL / PostgreSQL）。与本变更与 PostGIS 无关，不出 PostgreSQL+PostGIS 项。
- 开发库（实际为 PostgreSQL，见自动记忆 `backend-uses-postgresql`）的迁移执行由开发者决定时机，实现过程不擅自改库。

### 3.4 发布流程与不可逆（沿用现状）

- `POST /recipes/{id}/publish`（[recipes.py:589](backend/app/api/recipes.py#L589)）不变：管理员 `apply_as_admin` 直写、普通用户 `submit` 待审。
- 执行器 [recipe_publish.py](backend/app/services/proposals/executors/recipe_publish.py) 的 `revert` 保留（管理员救场）。
- 作者侧无「取消发布」入口 → 不可逆天然成立。
- 已发布菜谱删除限制（[recipes.py:635](backend/app/api/recipes.py#L635) 普通用户不可删）不变。

## 四、测试

- 后端单测（新增）：
  - 管理员创建菜谱默认 `is_public=False`
  - 普通用户创建默认私有
  - 导入路径产出 `is_public=True`
  - 列表/详情可见性：别人的 custom 私有菜谱看不到、导入菜谱看得到
  - 发布端到端：普通用户 submit 待审 / 管理员直写生效
- 回归：`test_permissions_p0` / `test_proposals_framework` / `test_recipes` / `test_recipe_edit_proposals`，修掉因判断逻辑由 `source` 改 `is_public` 而失败的旧用例（重点排查依赖「`source` 非空即公共」的断言）。
- 前端 `npm run build` 通过。
- 后端 `python -m compileall` 通过。

## 五、非目标（YAGNI）

- 不引入「取消发布 / 撤回公开」功能（作者侧不可逆）。
- 不清理 `source` 字段本身（保留作来源标识）。
- 不回收历史管理员创建的 custom 菜谱（`is_public=True` 保持）。
- 不改发布审核策略（仍 manual）。
- 不动 `RecipeResponse.source` 响应字段。

## 六、风险

- 判断侧改动点多（9 处），漏改一处会导致可见性/审核行为不一致 → 实现时用 grep 全量复核 `source != None` / `is_public_or_source` / `recipe.source is not None` 零残留。
- 导入菜谱数据迁移若漏跑，导入菜谱变私有 → 迁移 SQL 须在代码部署同步执行。
- 旧测试用例可能依赖 `source` 非空判公共 → 回归时逐一修正。
