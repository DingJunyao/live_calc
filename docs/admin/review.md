# 提议审核台

提议审核台是管理员管理共享数据改动的中心。普通用户对共享数据（原料、商品、单位、营养等）的修改，都会汇到这里等管理员拍板。

## 机制总览

生计的共享数据改动走"提议-审核"框架：

- **普通用户**改共享数据 → 生成一条**提议**（`change_proposals`，状态 `pending`）→ 管理员审核通过后才生效
- **管理员**改共享数据 → 走 `apply_as_admin`，**立即生效**并留痕（提议状态记为 `applied`）

原理见 [核心概念 · 数据归属](../concepts.md#g-数据归属谁的数据归谁)。

## 提议的生命周期

```
submit（提交，pending）
   │
   ▼
review（管理员审核）
   │
   ├── approve → applied（生效）
   │              │
   │              └── revert（回滚，恢复改动前）
   │
   └── reject → rejected（驳回）
```

## 审核策略

每类改动的审核策略由"治理总表"规定：

- **有数据**（改动已有内容）→ 人工审核（`manual`）
- **补空**（原本没数据、首次填入）→ 自动通过（`auto_approve`）或人工，取决于该类数据的策略
- **管理员直写**（`apply_as_admin`）→ 立即生效，绕过队列

## 审核台界面

审核台列表展示所有提议，可按状态筛（待审 / 已处理）。点进详情能看到：

- **实体标识**：可读名称（`entity_label`，如"原料：鸡蛋"），而不是干巴巴的 ID
- **变更 diff**：
  - 新增（`create`）：标"新增"，显示填了什么
  - 修改（`update`）：旧值 → 新值，高亮变化字段
  - 删除（`delete`）：显示被删对象的原始内容（删除提议的 payload 是空的，靠 `snapshot` 还原原内容）
- **操作**：批准 / 驳回 / 回滚

## 反级联回滚

删除类提议（删原料、删商品等）被批准后是**软删除**（标记 `is_active=False`）。如果后悔了，回滚会**反级联恢复**：

- 删原料 → 该原料的商品、价格记录都恢复了
- 删商品 → 该商品的价格记录恢复了

> 回滚依赖改动前的 `snapshot`，所以即使是删除，审核台也能看到"删了什么"并完整还原。

## 覆盖的改动类型

审核框架覆盖这些共享数据的改/删（对应各自的"执行器"）：

- **原料、商品**的改/删（含级联）
- **单位**的改/删（标准单位受保护，不可改基准/不可删）
- **实体单位覆盖**、**密度**
- **营养写入**、**USDA 匹配**
- **食材合并**、**食材层级**
- **商家**改/删
- **菜谱相关**（如菜谱图片删除等）

> 部分高危操作（如硬删除）保留管理员直写，不走提议；详情看各功能文档。

## 管理员直写

管理员修改上述数据时，系统用 `apply_as_admin` 走对应的执行器**立即应用**，同时在审核台留一条 `applied` 状态的提议（供追溯）。所以管理员的改动也留痕，只是不需要别人审。

## 提示

- 普通用户提交提议后，前端会提示"待审核"，**不会**误报"已保存"或"已删除"
- 提议生效前，对该用户来说改动还没落地（他看到的还是旧值，直到管理员批准）
- 管理员可在审核台一次处理多条提议
