# Vuetify 迁移设计方案 v2

## 概述

**目标**：完全重写前端，遵循 Material Design 3 交互逻辑，而非简单映射原有组件。

**核心原则**：
1. **交互优先**：先设计交互流程，再选择组件
2. **MD3 规范**：严格遵循 Material Design 3 设计规范
3. **业务驱动**：按业务优先级和依赖关系规划开发顺序
4. **移动优先**：优先考虑移动端体验，响应式适配桌面端

## 业务依赖关系

```
                    ┌─────────┐
                    │  商家   │
                    └────┬────┘
                         │
                         ▼
┌─────────┐        ┌─────────┐
│  商品   │───────→│ 价格记录 │
└────┬────┘        └─────────┘
     │
     │ 归属于
     ▼
┌─────────┐  包含/回退  ┌─────────┐
│  原料   │←───────────→│  原料   │
└────┬────┘             └─────────┘
     ↑                       ↑
     │ 1:1对应               │
     │                       │
┌─────────┐                  │
│ 菜谱食材│──────────────────┘
└────┬────┘
     │
     ▼
┌─────────┐
│  菜谱   │ ← 成本/营养计算（依赖原料层级关系）
└─────────┘
```

**关键关系说明**：
- **价格记录**：关联商品和商家（不直接关联原料）
- **商品**：归属于某个原料（商品是原料的具体实例，如"散装鸡蛋"归属于"鸡蛋"）
- **原料层级**：原料之间存在包含、回退等关系，影响菜谱成本和营养计算
- **菜谱食材**：每个食材**一对一对应**一个原料（包含数量、单位信息）
- **菜谱**：由多个食材组成，成本和营养值通过食材对应的原料及其层级关系计算

## 业务优先级

| 优先级 | 模块 | 说明 |
|--------|------|------|
| P0 | 价格记录 | 核心操作，用户最频繁使用；依赖商品和商家 |
| P1 | 菜谱管理及明细 | 核心功能，涉及成本/营养计算；依赖原料及层级关系 |
| P2 | 商品管理及明细 | 价格记录的前置依赖；归属于原料 |
| P2 | 原料管理及明细 | 菜谱/商品的前置依赖；包含层级关系管理 |
| P2 | 商家管理 | 价格记录的前置依赖 |
| P3 | 仪表盘 | 统计概览，优先级最低 |

### 响应式策略

| 断点 | 宽度 | 导航方式 | 布局 |
|------|------|----------|------|
| xs | < 600px | 底部导航栏 | 单列 |
| sm | 600-960px | 底部导航栏 | 单/双列 |
| md | 960-1280px | 侧边抽屉（可收起） | 双列 |
| lg/xl | > 1280px | 固定侧边栏 | 多列 |

### 移动端交互模式

**核心操作使用 FAB（悬浮按钮）**：
- 价格记录页面：右下角 FAB「添加记录」
- 菜谱列表页面：右下角 FAB「新建菜谱」
- 商品/原料列表：右下角 FAB「添加」

**表单使用全屏对话框**：
- 移动端表单 `<v-dialog fullscreen>`
- 桌面端表单 `<v-dialog max-width="500px">`

**列表使用无限滚动或分页**：
- 优先使用 `<v-infinite-scroll>`
- 大数据量使用 `<v-data-table-server>`

## 风险与缓解措施

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| ECharts 暗色模式 | 图表显示异常 | 封装组件，监听主题切换 |
| Leaflet 样式冲突 | 地图控件异常 | CSS 命名空间隔离 |
| 表单验证逻辑 | 迁移复杂 | 使用 VeeValidate 或 Vuetify 内置 |
| 移动端键盘 | 布局错乱 | 使用 Vuetify v-layout |
| 大数据量列表 | 性能问题 | 使用虚拟滚动 |
