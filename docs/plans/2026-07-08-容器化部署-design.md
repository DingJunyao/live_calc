# 容器化部署设计 · 生计

> 日期：2026-07-08
> 状态：设计稿（待实现）
> 来源：brainstorming 对话定稿

## 1. 目标

为「生计 - 生活成本计算器」提供两种 Docker 部署形态：

- **统一部署**：前后端打包进单个镜像（all-in-one），一个容器、一个端口即可访问完整应用
- **分开部署**：前端、后端各自独立镜像，便于水平扩展与负载均衡

两种形态均提供配置、数据库、静态文件、日志的持久化挂载点；同时镜像**开箱即用**——不挂任何挂载点也能启动，用于基础功能验证。

## 2. 决策摘要

| 决策 | 选型 | 理由 |
|---|---|---|
| 数据库 | 默认 SQLite，`DATABASE_URL` 走 `.env` | compose 不绑库；换 PG/MySQL 改一行即可（[requirements.txt](../../backend/requirements.txt) 已含 `psycopg2-binary` + `pymysql`） |
| 统一部署形态 | 单容器 nginx + uvicorn（supervisord） | 真 all-in-one；前端 nginx 配置可与分开部署复用 |
| 前端后端地址 | 运行时注入 `BACKEND_URL` | 前端镜像构建一次到处部署；`VITE_API_URL=/api/v1` 相对路径 |
| Agent（Claude Code CLI） | 不进镜像 | 镜像精简；维护任务台仅本机环境用 |
| HTTPS | 容器跑 HTTP，外层终结 TLS | 容器职责单一；续期交给云 LB / Cloudflare / 宿主反代 |

## 3. 镜像拓扑

一份根 `Dockerfile`，multi-stage 多 target，构出三种镜像：

```text
frontend-builder (node) ──► dist
        │
        ├──► frontend (nginx)            ← 分开部署用
        │
backend (python slim + uvicorn) ──────────► 分开部署用
        │
        └──► all-in-one (backend + dist + nginx + supervisord)  ← 统一部署用
```

三 target 共享前端构建层、后端依赖层，改一处三处同步，无复制粘贴。

## 4. 前端镜像

**`frontend-builder`**（`node:20-alpine`）
- `COPY package*.json` → `npm ci`（有 [package-lock.json](../../frontend/package-lock.json)，可复现）
- `COPY` 源码
- `ARG`/`ENV`：`VITE_API_URL=/api/v1`（相对路径）、`VITE_DATA_REPO_IMAGE_BASE`、`VITE_REQUEST_TIMEOUT`、`VITE_LONG_REQUEST_TIMEOUT`，默认对齐 [frontend/.env.example](../../frontend/.env.example)
- `npm run build` → `/app/dist`

**`frontend`**（`nginx:alpine`）
- 拷 `dist` → `/usr/share/nginx/html`
- 拷 `default.conf.template`（含 `${BACKEND_URL}` 占位）
- `entrypoint.sh`：`[ -f /app/.env ] && set -a && . /app/.env && set +a` → 取 `BACKEND_URL` → `envsubst` 渲染 → `exec nginx`
- nginx 三件事：`location /api/ { proxy_pass ${BACKEND_URL}; }` 反代后端；`location /` 静态 serve + `try_files $uri $uri/ /index.html`（vue-router history 兜底，否则刷新 404）；静态资源长缓存
- `EXPOSE 80`
- `BACKEND_URL` 默认值：split → `http://backend:8000`，all-in-one → `http://127.0.0.1:8000`

> 关键：`VITE_API_URL=/api/v1` 用相对路径 → 前端所有请求走「当前页面 origin + /api/v1」→ 由这层 nginx 反代到后端 → 同一个 dist 部署到任何域名/端口都对，后端地址完全由 nginx 决定。这是「构建一次，到处跑」的根基。

## 5. 后端镜像

**`backend`**（`python:3.11-slim`）
- `COPY requirements.txt` → `pip install --no-cache-dir -r requirements.txt`（分层缓存）
- requirements 已含 `psycopg2-binary` + `pymysql`，切 PG/MySQL 不改镜像
- `COPY app/ alembic/ scripts/`，`WORKDIR /app`
- `RUN mkdir -p data static logs`（开箱即用前提；虽然 [main.py](../../backend/app/main.py) 会建 static，logs 显式建更稳）
- 加非 root 用户跑（安全）
- `CMD uvicorn app.main:app --host 0.0.0.0 --port 8000`，不带 `--reload`（区别于开发的 `python -m app.main`）
- `EXPOSE 8000`；`HEALTHCHECK` 打 `/health`（用 `python -c "urllib.request.urlopen(...)"`，省装 curl）
- 不入镜像：`data/ logs/ .venv/ __pycache__/ tests/ .env`（`.dockerignore`）

## 6. 统一部署 · `docker-compose.yml`

```yaml
services:
  app:
    build: { context: ., target: all-in-one }
    image: livecalc:all-in-one
    ports: ["${APP_PORT:-80}:80"]
    env_file: [backend/.env]              # 后端配置（持久化在宿主文件；不存在需先 cp）
    volumes:
      - ./data:/app/data                  # SQLite 库 + 导入数据
      - ./static:/app/static              # 上传图片
      - ./logs:/app/logs                  # 日志
    restart: unless-stopped
```

all-in-one 内部：`supervisord` 同时管 nginx（80，serve 前端 dist + 反代 `/api` 到 `127.0.0.1:8000`）与 uvicorn（8000）。

## 7. 分开部署 · `docker-compose.split.yml`

```yaml
services:
  backend:
    build: { context: ., target: backend }
    image: livecalc:backend
    expose: ["8000"]                      # 仅内部网络，不映射宿主
    env_file: [backend/.env]
    volumes:
      - ./data:/app/data
      - ./static:/app/static
      - ./logs:/app/logs
    restart: unless-stopped
  frontend:
    build: { context: ., target: frontend }
    image: livecalc:frontend
    ports: ["${APP_PORT:-80}:80"]
    env_file: [frontend/.env]             # 含 BACKEND_URL；脱 compose 时改后端地址
    depends_on:
      backend: { condition: service_healthy }
    restart: unless-stopped
```

脱 compose 裸部署时，`BACKEND_URL` 改成 `http://远端IP:8000`，前端镜像一个不动。

## 8. `.env` 处理策略

**核心原则：所有挂载点（含 `.env`）都是「挂了用你的，不挂用镜像内置默认」。**

### 后端（运行时配）
- compose `env_file: [backend/.env]` 把宿主文件注入为容器环境变量（pydantic 环境变量优先级最高，容器内无需 `.env` 文件）
- 不挂：用 [config.py](../../backend/app/config.py) 字段默认值兜底（`DATABASE_URL`、`APP_HOST/PORT`、JWT dev 密钥等均齐全）
- 为何不用 `volume: ./backend/.env:/app/.env`：宿主文件不存在时 docker 会挂成空目录、容器内 `/app/.env` 变目录致 pydantic 报错；`env_file` 避此坑
- `.env` 不入镜像（`.dockerignore`）
- 部署：`cp backend/.env.example backend/.env` 改值

### 前端（构建期 + 运行时分工）
- **构建期 ARG**（编译进 JS，不变）：`VITE_API_URL`、`VITE_DATA_REPO_IMAGE_BASE`、`VITE_REQUEST_TIMEOUT`、`VITE_LONG_REQUEST_TIMEOUT`；Dockerfile 里 `ARG → ENV` 让 vite 构建时读到
- **运行时**（会变）：`BACKEND_URL`，compose `env_file: [frontend/.env]` 注入（`.env` 里其余 `VITE_*` 是开发变量，运行时容器沉默无害），entrypoint 读取后 envsubst
- [frontend/.env.example](../../frontend/.env.example) 补一行 `BACKEND_URL=http://backend:8000`（标注「部署用」），与开发的 `VITE_*` 共存一份
- `VITE_DEV_PORT` / `VITE_DEV_BACKEND_URL` 是 dev 用的，生产构建不传
- 原则一句话：**随部署环境变的走运行时，跟代码行为一样不变的走构建期**

### 敏感值
`JWT_SECRET_KEY` 等开箱默认是 dev 占位，**生产挂 `.env` 时必须改**（文档标红）。

## 9. 持久化挂载点

| 挂载 | 容器路径 | 用途 | 不挂后果 |
|---|---|---|---|
| `backend/.env` | `env_file` 注入 | 后端配置 | Settings 字段默认值兜底（可跑） |
| `frontend/.env` | `env_file` 注入 | 前端 `BACKEND_URL` | entrypoint 默认值兜底（可跑） |
| `./data` | `/app/data` | SQLite 库 + 导入数据 + 备份 | 容器内 ephemeral，重启丢库（致命） |
| `./static` | `/app/static` | 上传图片 | 重启丢图 |
| `./logs` | `/app/logs` | 运行日志 | 容器内写，删容器丢（可接受） |

**注**：`data/static/logs` 是 bind volume 挂载点；`.env` 用 `env_file` 注入——避免 volume 挂文件时「宿主不存在 → 被当目录挂载 → 容器内路径变目录」的坑。两种方式都把配置持久化在宿主文件。

## 10. 开箱即用机制

裸 `docker run` 不挂任何东西即可验证基础功能：

```bash
docker run -p 80:80 livecalc:all-in-one
```

兜底链：
- **后端**：Settings 字段默认值 + Dockerfile `mkdir data static logs` + 首启 `create_all` 建库 + `init_default_data` 建单位/分类
- **前端**：entrypoint `BACKEND_URL` 默认值 + envsubst 渲染
- **all-in-one**：单命令即完整应用（SQLite ephemeral + 默认配置 + 首启导入菜谱）

> 首启导入：`FIRST_RUN_INIT_RECIPES` 默认 `true` → 从 GitHub clone HowToCook 数据（首启慢、需网络；之后跳过）。离线/嫌慢在 `.env` 设 `false`，或用 `DATA_LOCAL_PATH` 挂本地数据。

## 13. 使用说明

**构建：**
```bash
# 统一部署
docker compose build
# 分开部署
docker compose -f docker-compose.split.yml build
# 或按 target 单独构
docker build --target all-in-one -t livecalc:all-in-one .
docker build --target backend   -t livecalc:backend   .
docker build --target frontend  -t livecalc:frontend  .
```

**运行：**
```bash
# 统一部署（compose，持久化）
docker compose up
# 分开部署（compose，持久化）
docker compose -f docker-compose.split.yml up
# 开箱验证（裸 run，ephemeral）
docker run -p 80:80 livecalc:all-in-one
```

## 14. 边界与注意事项

- **负载均衡**：`backend` 多副本（`scale`）需切 PG（SQLite 单文件多写不支持）；前端 `BACKEND_URL` 指单上游或外层 LB。本期 compose 默认单副本。
- **HTTPS**：容器仅 HTTP，TLS 由外层终结；如需容器内证书（Caddy / certbot），另开一期。
- **首启导入**：默认联网拉菜谱，离线设 `FIRST_RUN_INIT_RECIPES=false`。
- **生产密钥**：`JWT_SECRET_KEY` 等必须改，勿用 dev 占位。
- **Agent 任务台**：生产镜像不含 Claude Code CLI，功能不可用；本机环境用。
- **数据库迁移**：项目不走 alembic（`create_all` 建库）；切 PG/MySQL 需跑 [scripts/sql](../../backend/scripts/sql) 初始化脚本。
- **时区**：容器默认 UTC；应用层已统一「用户本地日」（见 [FEATURE_时间口径统一为用户本地日](../../cc/FEATURE_时间口径统一为用户本地日.md)），前端注入 `X-Timezone` header，无需容器层处理。
