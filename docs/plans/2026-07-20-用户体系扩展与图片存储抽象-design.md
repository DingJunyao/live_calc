# 用户体系扩展与图片存储抽象·设计文档

> 日期：2026-07-20
> 状态：设计稿（待评审）
> 范围：用户信息扩展（头像 / 昵称 / 地区 / 第三方登录预留）+ 图片存储抽象（本地 / S3 兼容）

---

## 一、背景与目标

### 背景

项目当前状态：

- **用户体系**仅有 username / email / phone / password，缺头像、昵称、地区；无第三方登录能力。
- **图片**只存本地 `backend/static/images/recipes/`，零存储抽象。`static` 根路径在 9 处各自计算，`/static/images/recipes/` 字面量在 10 处硬编码散落。

### 目标

1. **用户信息扩展**：加头像、昵称、地区；为第三方登录预留通用 provider 框架。
2. **图片存储抽象**：引入 `StorageBackend`，兼容本地与 S3（含阿里云 OSS / 腾讯云 COS / MinIO）；初始化、上传、清理三个环节同步接入。

### 非目标（YAGNI）

- 第三方登录流程（`wx.login` / OAuth code 交换等）本次**不实现**，仅预留表结构与字段。
- 全球行政区划细化数据本次**不入库**（仅中国县级 + 全球国家级打底）。
- 商品图上传端点本次不做（`Product.image_url` 字段已有，消费侧已支持）。
- 不重写第三方 SDK 原生客户端，统一走 S3 兼容协议。

## 二、现状摸底（关键发现）

### 2.1 用户与认证

- [user.py](backend/app/models/user.py) 已有 `phone`（unique nullable）、`token_version`（密码重置失效机制）；缺 avatar / nickname / 地区 / 第三方关联。
- JWT 已有 `ver` claim + `token_version` 机制（[security.py:139](backend/app/core/security.py#L139)），将来可复用于 OAuth 绑定/解绑。
- 前端 [types/index.ts:3](frontend/src/types/index.ts#L3) 的 `User` 接口含幽灵字段 `is_superuser`（后端无此列），且缺 phone / is_active / email_verified / nutrition_goals / daily_budget / unit_preferences 等真字段，代码靠 `as any` 绕过。
- [auth.py](backend/app/api/auth.py) 里 `UserResponse` 手工拼了 **7 处**（`_user_to_response` helper 已抽但未全用）。
- [region_unit_setting.py](backend/app/models/region_unit_setting.py) 是**国家级单位偏好表**（region_code 如 "CN"，~1 条），**不是行政区划树**；其 `UserUnitPreference` 表与 `User` 表上的单位字段语义重复（疑似早期遗留，本次不动）。

### 2.2 图片存储

- 唯一图片上传端点 `POST /recipes/{id}/images`（[recipes.py:650](backend/app/api/recipes.py#L650)）；原料无图、商品 `image_url` 字段在但无上传端点、无头像。
- 三个 importer 的 `_download_image` fallback **不一致**：`JsonRecipeImportService` 失败跳过，`enhanced` / `howtocook` 两个仍写残缺 `images/xxx.jpg`。
- **清理三缺口**：① update 菜谱走通用 setattr，移出列表的图不 unlink；② 软删菜谱不动图；③ 无定时孤儿清理。
- 商品 `image_url` 导入未过 `_restore_image_paths`（[export.py:523](backend/app/services/importer/importers/export.py#L523)），残留缺前缀路径。
- 前端 [image.ts:14](frontend/src/utils/image.ts#L14) 兜底默认指向 GitHub `corr` 分支，与后端 `data_repo_branch` 默认 `main` 不一致。

## 三、总体路线图

三阶段，依赖关系如下：

```
阶段一（图片存储抽象 · 地基）
    │
    └──→ 阶段二·A 的 avatar 上传（强依赖）

阶段二·A 的 nickname / 类型修复   ─┐
阶段二·B 行政区划                ├── 互不依赖，三者可并行
阶段三 OAuth 预留                ─┘
```

- **地基先行**：阶段一不依赖任何东西，是串行起点。
- **avatar 是交叉点**：阶段二·A 的头像上传强依赖阶段一；nickname 改动、前端类型修复不依赖。
- **OAuth 独立**：阶段三仅建表，与其它阶段零耦合。

## 四、阶段一·图片存储抽象层

### 4.1 StorageBackend 接口

新建 `backend/app/services/storage/`，定义统一契约：

```python
class StorageBackend(Protocol):
    def put(self, key: str, data: bytes, content_type: str) -> str:
        """存储，返回 key"""
    def get(self, key: str) -> bytes:
        """读取"""
    def delete(self, key: str) -> None:
        """删除"""
    def exists(self, key: str) -> bool:
        """存在性"""
    def url_for(self, key: str) -> str:
        """对外可访问 URL"""
```

### 4.2 两个实现

- **LocalBackend**：落 `backend/static/images/<目录>/`，`url_for` 返 `/api/v1/static/...`（沿用现有 StaticFiles 挂载）。
- **S3Backend**：boto3 走 S3 兼容端点（OSS / COS / MinIO / AWS 通吃），`url_for` 返永久公共 URL。

### 4.3 DB 存储范式

统一存 **key**（相对路径，如 `recipes/xxx.jpg`、`avatars/yyy.png`），**不存完整 URL**。切 backend 时 DB 零改动，`url_for` 动态拼。现有 `/static/images/recipes/xxx.jpg` 迁移时剥前缀成 key。

### 4.4 三环节收口

**上传**：

- 现有 `recipes/images` 端点 → 全走 `backend.put`。
- 新增 `POST /auth/me/avatar`（阶段二·A）→ `backend.put('avatars/...', ...)`；换头像时 `backend.delete` 旧头像。

**初始化**：

- 三个 importer 的 `_download_image` 统一走 `backend.put`。
- 统一 fallback 策略：**失败跳过，不写残缺路径**（对齐 `JsonRecipeImportService`）。

**清理**：

- 孤儿清理、update 移出图、软删菜谱 → 全走 `backend.delete`。
- 补 CLAUDE.md 点名的缺口：菜谱 update 时 diff 旧图列表，对移出项调 `delete`。

### 4.5 顺手治的 bug

- 三个 importer 残缺路径 fallback 统一（`enhanced` / `howtocook` 对齐 `json`）。
- 商品 `image_url` 导入过 `_restore_image_paths`（[export.py:523](backend/app/services/importer/importers/export.py#L523)）。
- [admin.py:277](backend/app/api/admin.py#L277) 孤儿列表 `url` 补 `/api/v1` 前缀（或统一走抽象层）。
- 前端 [image.ts:14](frontend/src/utils/image.ts#L14) 兜底分支默认与后端 `main` 对齐（或移除 GitHub 兜底，统一走 storage）。

### 4.6 迁移脚本

- `storage.migrate_local_to_s3()`：批量读 local 文件 → `S3Backend.put` → 校验。
- **本次交付**（用户拿真实 OSS bucket 自测）。

### 4.7 配置项

`Settings` 新增（均给默认值，local 模式下 `s3_*` 可空）：

- `storage_backend: 'local' | 's3'`（默认 `'local'`）
- `storage_base_url`（local 端用）
- `s3_endpoint` / `s3_access_key` / `s3_secret_key` / `s3_bucket` / `s3_region`

## 五、阶段二·A·用户字段扩展

## 六、阶段二·B·行政区划模块

### 6.1 表结构

```python
class AdministrativeRegion(Base):
    __tablename__ = "administrative_regions"
    id = Column(Integer, primary_key=True, index=True)
    code = Column(String(12), unique=True, nullable=False)   # 行政区划编码
    name = Column(String(100), nullable=False)
    name_en = Column(String(100), nullable=True)             # 国际化
    parent_id = Column(Integer, ForeignKey("administrative_regions.id"), nullable=True)
    level = Column(Integer, nullable=False)                  # 0国 1省 2市 3县
    iso_country = Column(String(2), nullable=True)           # ISO 3166-1
    path = Column(String(200), nullable=True)                # 如 CN/110000/110100
    is_active = Column(Boolean, default=True)
    created_at = Column(DateTime(timezone=True), server_default=func.now())
    updated_at = Column(DateTime(timezone=True), onupdate=func.now())
```

### 6.2 数据范围与来源

- **本次入库**：中国省/市/县全套（~3500，国家统计局标准码）+ 全球国家级（~250，ISO 3166-1）。
- **数据源**：开源包（如 `modood/Administrative-divisions-of-China`）打包进 `backend/scripts/` 或 `data/regions/`。
- **维护**：lifespan `ensure_administrative_regions`（幂等，count>0 跳过），类似 [allergen_seed.py](backend/app/services/allergen_seed.py)。
- **按需补**：导入机制设计成「按国家数据包补」——将来开拓某国市场，加该国数据包重跑 ensure。

### 6.3 User 字段

- 加 `region_id`（FK → administrative_regions.id，nullable）。
- 用户选到任意层级都行：中国可到县，国外暂到国家。

### 6.4 API

- `GET /regions?parent_id=&level=`：懒加载子节点（前端级联用，不一次性吐几千条）。
- `GET /regions/{id}`：返含完整 path 的节点（展示「中国/广东省/深圳市/南山区」）。

### 6.5 前端

- 级联选择器（按 parent 懒加载），嵌进 ProfileView 编辑对话框。国外用户选到国家即停。

### 6.6 与现有 region_unit_setting 的关系

- **不合并**：职责不同（单位偏好配置 vs 地理树）。
- 国家级节点用同一套 ISO 3166-1 码，将来可自然关联。

## 七、阶段三·OAuth 通用 provider 预留

### 7.1 表结构（本次新建）

```python
class UserOauthAccount(Base):
    __tablename__ = "user_oauth_accounts"
    id = Column(Integer, primary_key=True, index=True)
    user_id = Column(Integer, ForeignKey("users.id"), nullable=False)
    provider = Column(String(32), nullable=False)        # wechat_mp / wechat_web / alipay / google / github
    provider_user_id = Column(String(128), nullable=False)  # openid
    unionid = Column(String(128), nullable=True)         # 微信体系打通
    access_token = Column(String(512), nullable=True)    # 远期对接用，本次留空
    refresh_token = Column(String(512), nullable=True)
    expires_at = Column(DateTime(timezone=True), nullable=True)
    created_at = Column(DateTime(timezone=True), server_default=func.now())
    updated_at = Column(DateTime(timezone=True), onupdate=func.now())
    __table_args__ = (
        UniqueConstraint("provider", "provider_user_id", name="uq_provider_user"),
        Index("ix_oauth_unionid", "unionid"),
    )
```

### 7.2 User 表零改动

- `username` / `email` / `password_hash` **全保持 NOT NULL**。
- 不引入「无密码账号」新状态，登录 / JWT / 改密码逻辑**不动**。

### 7.3 将来业务语义（本次不实现）

- 第三方发起注册：点 OAuth 登录 → 识别 openid 未绑定 → 引导补填 username / email / 密码 → 建完整本地账号 + 绑定 OAuth → 签发 token。
- 本质：OAuth 是**注册入口**，本地账号仍是**主体**。

### 7.4 为什么是轻预留

- 工具类应用，用户不为「简化注册」买单，数据完整性优先。
- 将来若需放宽（无密码账号），因 OAuth 表独立设计、不绑死 User 结构，改动可控。

## 十、关键决策记录

| 决策点 | 选择 | 理由 |
|---|---|---|
| 路线图 | 三阶段，地基先行 | avatar 依赖存储抽象 |
| OAuth 表 | 通用 provider 框架 | 一劳永逸，多平台复用 |
| 账号关联 | 第三方直登，自动建号 | 现代主流入口 |
| username / email / password | 全保持 NOT NULL | 数据完整性优先 |
| OAuth 本次范围 | 仅建表，业务逻辑远期 | 工具类应用不需简化注册 |
| 图片 DB 范式 | 存 key 不存 URL | 切 backend 零改动 |
| bucket 可见性 | 公共读（永久 URL） | 菜谱图 / 头像本就公开 |
| S3 实现 | S3 兼容 backend（boto3） | 一套 SDK 通吃 OSS / COS / MinIO |
| 行政区划范围 | 中国县级 + 全球国家级 | 数据量可控，结构支持按需补 |
| S3 验收 | 用户拿 OSS bucket 自测 | 不依赖外部资源阻塞开发 |
