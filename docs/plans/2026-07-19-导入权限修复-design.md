# 导入权限修复设计

- 日期：2026-07-19
- 状态：设计稿（待实现）
- 关联：[cc/MULTI_USER_PERMISSIONS.md](../../cc/MULTI_USER_PERMISSIONS.md)、[cc/FEATURE_数据导出.md](../../cc/FEATURE_数据导出.md)、[cc/BUGFIX_黑名单分组导入缺失.md](../../cc/BUGFIX_黑名单分组导入缺失.md)

## 1. 背景与问题

多用户权限系统上线后，数据导出有两档：

- `full`：仅管理员，导出全表（含他人私有数据、管理员维护的全局结构）
- `mine`：普通用户，自己的数据 + 沿外键可达的共享字典（保证引用完整）

但数据导入这条通道完全没有按角色收口。实测：管理员导出 full 包，普通用户导入后突破了普通用户的所有限制——

- 普通用户凭空创建了管理员专属的黑名单分组定义（API 端这些操作全部 `get_current_admin_user`）
- 普通用户导入即获得 `is_public=True` 的菜谱，绕过发布审核工作流
- 管理员的个人行为数据（价格记录、家/公司地点、个人黑名单、分组订阅）被克隆到普通用户名下，污染其成本计算与个人视图
- 共享字典（原料/商品/商家/营养）被以普通用户身份直写，绕过提议-审核框架

## 2. 根因

三层缺口，逐层证据：

- **入口** [import_api.py:33](../../backend/app/api/import_api.py#L33) `POST /data/upload` 用 `get_current_user`——设计意图是保留普通用户导入「系统导出格式」的能力，但「系统导出格式」恰恰是最敏感的全量数据格式
- **闸门** [api_service.py:209-230](../../backend/app/services/importer/api_service.py#L209) `import_from_upload_path` 只对 HowToCook 格式拦普通用户（L216-217），EXPORT 格式完全放开
- **漏洞主体** [export.py](../../backend/app/services/importer/importers/export.py) `ExportImporter`：仅依赖 `self.user_id`，`grep "is_admin|admin|current_user|permission"` 0 命中；菜谱 `is_public=True` 硬编码（L365）；admin-only 数据（黑名单分组 L610、单位换算 L148、商品条码 L469、营养 L292）一律裸 `db.add`

## 3. 设计目标

1. **堵住越权**：普通用户导入不再能创建 admin-only 数据、不再绕过菜谱发布审核、不再克隆他人隐私
2. **保留迁移能力**：普通用户仍可导入 full 包作为「种子/迁移」、可导入 mine 包作为「备份恢复」
3. **权限一致性**：导入路径的处理与对应 API 端点权限规则对齐——同一份数据，从 UI 建和从导入建，权限边界相同
4. **附带补全数据丢失**：商品字段、entity_unit_overrides 表（调查中发现的无关权限的 bug）

## 4. 方案总则

给 `ExportImporter` 注入 `is_admin` 标志 + 从 manifest 读 `scope`，导入时按「角色 × scope × 数据桶」三维分流：

- **管理员**导入任意包：维持现状，全量直写（含 `is_public`、`is_verified` 原值）
- **普通用户**导入：按下表分桶处理
- **入口不动**：`POST /data/upload` 仍允许普通用户调（保留迁移能力），权限收口放在 importer 内部

`scope` 来源：[packaging.py:393](../../backend/app/services/export/packaging.py#L393) manifest 已含 `"scope"` 字段。导入时由 `ExportImporter` 从 manifest 读取。

## 5. 分流矩阵（普通用户）

| 数据桶 | 代表表 | full 包 | mine 包 | 依据 |
|---|---|---|---|---|
| admin-only·结构 | `BlacklistGroup` / `BlacklistGroupIngredient` | 跳过 | 跳过 | API 全 admin-only |
| admin-only·字典 | `UnitConversion` / `ProductBarcode` | 跳过 | 跳过 | API admin-only |
| 共享字典 | `Unit` / `IngredientCategory` / `IngredientHierarchy` / `EntityDensity` / `Ingredient` / `Product` / `Merchant` | 复用或新建（`created_by`=当前用户） | 复用或新建 | 新增=auto 级；接受轻微越权（数据源是管理员授权包），不走提议以免爆审核台 |
| 营养数据 | `NutritionData` | 复用或新建，**`is_verified` 强制 False**、source 标 import | 同左 | 普通用户无权自证核验 |
| 菜谱 | `Recipe` | **强制 `is_public=False`**，作者=当前用户 | 恢复原 `is_public` | full 不绕过审核；mine 是自己的备份 |
| 个人行为 | `ProductRecord` / `UserPlace` / `UserIngredientBlacklist` / `BlacklistGroupSubscription` | **跳过** | 恢复（改归属=当前用户） | full 是别人隐私；mine 是自己的 |
| 商品·营养内嵌 | `Product.custom_nutrition_data` | 随商品复用或新建，按营养数据同款降级 | 同左 | 与 NutritionData 口径一致 |

说明：

- admin-only 数据即使在 mine 包里也跳过——它们本就不归个人所有（mine 包里若有也是引用快照，不应重建为个人数据）
- 「复用或新建」语义：按业务键（name 等）查含软删，已存在则登记 id 映射不重建，不存在则新建（`created_by`=当前用户，视为 auto_approve 级）
- 「跳过」不抛错、不中断导入，计入 `skipped` 统计

## 6. 实现要点

### 6.1 is_admin 透传链路

- [api_service.py](../../backend/app/services/importer/api_service.py) `import_from_upload_path` 已接收 `is_admin`，透传给 `ExportImporter.__init__`：`ExportImporter(db, user_id=user_id)` → `ExportImporter(db, user_id=user_id, is_admin=is_admin)`
- `ExportImporter.__init__(self, db, user_id, is_admin=False)` 存 `self.is_admin`
- 其它入口（`import_from_git` / `import_from_local`）本就 admin-only，传 `is_admin=True`

### 6.2 scope 读取

- `import_all` 开头读 manifest 的 `scope` 字段；包无 scope 字段时按 `full` 处理（保守，取最严限制）
- 若 collection 已携带 manifest 则直接用，否则在 importer 内单独读 zip 内 `manifest.json`

### 6.3 各 `_import_*` 方法改造

按分流矩阵，为每个方法加普通用户分支。重点方法：

- `_import_blacklist_groups`（L588）：非 admin → 跳过整组（含 BlacklistGroupIngredient）
- `_import_unit_conversions`（L148）：非 admin → 跳过
- `_import_product_barcodes`（L469）：非 admin → 跳过
- `_import_nutritions`（L292）：非 admin → `is_verified=False`、source 标记 import
- `_import_recipes`（L341）：非 admin 时，`is_public = (scope=='mine') and 原值`；full 强制 False
- `_import_price_records` / `_import_user_places` / `_import_user_blacklist` / `_import_blacklist_subscriptions`：非 admin 且 scope=full → 跳过；scope=mine → 维持当前改归属行为
- `_import_products`（L409）：非 admin 时，`custom_nutrition_data` 跟营养数据同款降级（剥离 is_verified 语义或标记 source）

### 6.4 skipped 统计

- [models.py](../../backend/app/services/importer/models.py) `ImportResult` 加 `skipped: dict[str, int]`（key 为桶/原因，如 `blacklist_groups`、`price_records_other_user`）
- 前端 [ImportUploadDialog.vue](../../frontend/src/components/import/ImportUploadDialog.vue) 结果展示 skipped 摘要

### 6.5 附带修复（数据丢失，与权限无关）

- `_import_products`：补读 `barcode` / `image_url` / `custom_nutrition_data` / `custom_nutrition_source`（对齐 [serializers.py:309-326](../../backend/app/services/export/serializers.py#L309) 导出字段）
- 新增 `_import_entity_unit_overrides` 方法并在 `import_all` 注册（对齐 [packaging.py:445](../../backend/app/services/export/packaging.py#L445) 导出已含此表）

## 7. 用户反馈

导入完成后，结果统计区分三类：

- `imported`：成功导入数（按桶）
- `skipped`：跳过数 + 原因（admin-only / 他人隐私 / 已存在复用）
- `errors`：失败明细

前端 `ImportUploadDialog` 展示 skipped 摘要，让用户知道「这个 full 包里的管理员数据和他人隐私没有导入进来」，避免「怎么少了一截」的困惑。

## 8. 测试策略

- **单元测试**：`ExportImporter` 各桶在三种情形下的行为
  - （普通用户 × full）：黑名单分组不新增、菜谱 is_public=False、价格记录不克隆、营养 is_verified=False
  - （普通用户 × mine）：菜谱恢复原 is_public、价格记录恢复改归属、私人数据完整回来
  - （管理员 × 任意）：全量原样（回归现状）
- **端到端**：管理员导出 full → 普通用户导入 → 断言无越权数据、无他人隐私克隆、skipped 统计准确
- **回归**：mine 包自导自导回，数据完整；管理员 full 导入行为不变
- **附带**：商品 barcode/image_url/custom_nutrition_data 导入后落库；entity_unit_overrides 导入后落库

## 9. 不在本次范围

- HowToCook 格式导入（本就 admin-only，无需改）
- 导出端（导出权限已正确，无需改）
- change_proposals 框架本身
- 用户偏好类表（`UserProductWeightOverride` / `UserMerchantFavorite` / `UserMerchantProductOrder`）的导入——导出/导入当前均未涵盖，另议
- entity_unit_overrides 写入权限语义（本次只补导入落库，写入权限仍走对应 API 分流）

## 10. 决策记录

本次 brainstorming 五个关键决策（含理由）：

1. **整体方向：两者兼顾**——既要堵越权又要保留迁移能力，按权限逐桶精细处理（非简单拒绝）
2. **共享字典不存在时：直写新建**——视为 auto_approve 级，数据源是管理员授权包，可接受轻微越权；不爆审核台
3. **私人数据 full 包：跳过 / mine 包：恢复**——价格记录会污染成本计算、地点含坐标隐私，full 不克隆
4. **营养 is_verified：普通用户导入强制降级 False**——避免借导入自证核验
5. **mine 包菜谱：恢复原 is_public**——自己的备份恢复原状，本人发布的菜谱搬回来保持 public
