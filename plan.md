我希望能够创建一个基于 Web 的项目，也有移动端 APP 的计划，但目前还是重点在 Web 上。

项目名称：生计。意思就是“生活成本计算器”。

该项目可以：

- 记录自己从不同时间、不同地方的商店 / 超市 / 市场购买的商品的价格、重量、单价等信息
- 据上面的信息，计算烹饪一道菜所需食材的成本，估计其营养成分
- 向用户推荐经济实惠、营养均衡的购买、烹饪方案（比如午餐和晚餐做什么菜、这些菜涉及到的食材从哪里买更实惠——还要考虑交通成本）
- 生成每日、每周、每月的生活成本报告，帮助用户更好地管理自己的开支。

数据库支持 SQLite、MySQL、PostgreSQL，在兼容的基础上，充分利用它们的特性。

Web 前端需要支持 PWA，需要考虑移动端的使用体验。

可以开放注册，也可以通过邀请码注册。注册时需要填写用户名、密码、邮箱、手机号码等信息。登录和注册时，密码从客户端就要用 SHA256 加密。注意，需要同时支持 HTTP 和 HTTPS。

营养成分的计算，可以参考 USDA 的食品营养数据库。但是我不知道如何很好地集成这个数据库，以及如何很好地配合用户写的食材名称，希望你能帮我想一个方案。

价格需要支持多币种，单位需要支持多种单位（公制 / 市制 / 英制（甚至有些不同地区的市制和英制都不一样）），以及之间的转换。

对于购买地点，用户需要录入：

- 购买地点名称
- 购买地点地址（可选）
- 购买地点经纬度（可选，但是这个一般来说根据地址生成，要么就是用户在地图上选择，反正不是外显的）

对于记录商品价格，用户需要录入：

- 商品名称
- 购买时间
- 购买地点
- 价格
- 重量 / 数量
- 其他备注信息（可选）

用户也许只是记录单价。这时候就是：

- 商品名称
- 记录时间
- 记录地点
- 单价
- 单位
- 其他备注信息（可选）

当然，这两种情况放到一张表就行了，记一下是实际购买还是记录。但是页面上要提供便捷的记录方式。

菜谱可以参考：https://github.com/Anduin2017/HowToCook 。用户还可以录入自己烹饪的菜肴信息。

除外，还有水电气的费用、交通费用也要考虑。但是我没有什么太好的方案。

需要集成地图服务（高德 / 百度 / 腾讯 / 天地图 / OpenStreetMap），来展示用户添加的商店、超市等信息。要计算用户从家 / 工作地到达这些地方需要花费的的时间和成本。
